## 2. SPLとは何かとSolana CLIとの関係

### 2.1 SPLは何の略か

`SPL`は**Solana Program Library**の略で、Solana上で広く使われる標準プログラム群を指す。  
この中に、FT（Fungible Token）を扱うための**SPL Token Program**が含まれている。

### 2.2 「SPLトークン」と呼ばれる理由

一般に「SPLトークン」とは、SPL Token Programの仕様に従って作られたトークンを意味する。  
EthereumでいうERC-20トークンに近い立ち位置だが、実装モデルは異なる（SolanaはMint + Token Accountモデル）。

### 2.3 Solana CLIと`spl-token`の関係

ここは混同しやすいため、役割を分けて整理する。

| 要素 | 正体 | 主な役割 |
|------|------|----------|
| `solana` コマンド | Solana CLI本体 | ネットワーク設定、SOL残高確認、送金、キーペア管理など |
| `spl-token` コマンド | SPL Token用CLIツール（`spl-token-cli`） | Mint作成、トークンアカウント作成、Mint/Burn/Transferなど |
| `Tokenkeg...` | オンチェーンのSPL Token Program ID | 実際にトークン操作ロジックを実行するプログラム |

重要なのは、`solana`や`spl-token`は「ローカルで使う操作ツール」、`Tokenkeg...`は「チェーン上で動く実行主体」という点である。

### 2.4 なぜ`solana`だけでなく`spl-token`も使うのか

`solana`コマンドは汎用的なCLIであり、SPLトークン専用の高レベル操作は`spl-token`の担当範囲となる。  
そのため、トークン開発では通常次の使い分けを行う。

- ネットワークやウォレットの管理: `solana`
- トークンの発行・焼却・送金・供給量確認: `spl-token`

### 2.5 「標準で入っているのか」の整理

- **SPL Token Program**はSolanaの標準的なオンチェーンプログラム
- **`spl-token`コマンド**はローカルツールで、環境により導入済み/未導入がある

本日の環境で`spl-token create-token`が実行できたことから、`spl-token-cli`は利用可能状態である。

## 5. SPLトークンの作成と意味の読み解き

### 5.1 Mintの作成

```bash
spl-token create-token
```

代表的な出力項目の意味:

| 項目 | 意味 |
|------|------|
| `Address` | 作成されたMintアドレス（トークンのID） |
| `Decimals` | 小数桁数（例: 9なら最小単位は10^-9） |
| `Signature` | トランザクション署名（Tx ID） |
| `Tokenkeg...` | 標準SPL Token ProgramのプログラムID |

`TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA`は、トークン機能を提供する標準プログラムの識別子であり、作成したトークン自体のアドレスではない。

### 5.2 トークンアカウント作成

```bash
spl-token create-account <MINTアドレス>
```

このコマンドで作られるのは「そのMintを保管する口座（Token Account）」であり、残高はこの口座に記録される。

## 6. Mint・Supply・Balance・Burnの実践

### 6.1 発行前後の残高確認とMint

```bash
spl-token balance <MINTアドレス>
spl-token mint <MINTアドレス> 1000
spl-token balance <MINTアドレス>
```

### 6.2 `balance` と `supply` の違い

```bash
spl-token balance <MINTアドレス>
spl-token supply <MINTアドレス>
```

- `balance`: 現在ウォレット（または指定owner）が保有する量
- `supply`: そのMint全体の総発行量

### 6.3 Mint Authorityと発行可能者

「誰がMintできるか」はMintアカウントの`Mint Authority`で決まる。通常は`create-token`実行時に署名したキーペアが設定される。

### 6.4 総発行量の固定（Mint権限の無効化）

```bash
spl-token authorize <MINTアドレス> mint --disable
```

これ以降は追加Mintが失敗し、実質的に総供給量が固定される。

### 6.5 Burn時にToken Accountを指定する理由

```bash
spl-token burn <TOKENアカウント> 100
```

Burnは「どの口座の残高を減らすか」を特定する操作のため、MintアドレスではなくToken Accountを指定する。Mintは通貨の定義情報であり、保有残高そのものはToken Account側に存在する。

## 7. ウォレットアドレス・Token Account・Mintの違い

### 7.1 Solana内部での役割

| 要素 | 役割 | 例 |
|------|------|----|
| ウォレットアドレス | 署名主体（所有者） | Aさん/Bさん |
| Mintアドレス | トークン種別のID | Xトークン |
| Token Account | そのMintの保有残高を持つ口座 | AのX口座、BのX口座 |

同じMintをAとBが保有する場合、通常はA用とB用で別々のToken Accountが存在する（アドレスも異なる）。

### 7.2 SolanaとEthereumの比較

| 観点 | Solana | Ethereum |
|------|--------|----------|
| トークン識別子 | Mintアドレス | ERC-20コントラクトアドレス |
| 残高の保存先 | 各ユーザーのToken Account | ERC-20内部の`mapping(address => balance)` |
| 送金時に本質的に必要な宛先 | 宛先Token Account | 宛先EOA/コントラクト（内部台帳更新） |
| ユーザー識別子 | ウォレット公開鍵 | EOAアドレス |

要点として、Ethereumは「コントラクト内台帳モデル」、Solanaは「口座分離モデル（Mint + Token Account）」である。

## 8. 送金時にウォレットアドレスだけで送れる仕組み

### 8.1 基本コマンド

```bash
spl-token transfer <MINTアドレス> <数量> <受取人ウォレットアドレス> --fund-recipient
```

### 8.2 仕組みの実態

厳密には、SPLトークン送金の宛先はToken Accountである。ただしCLIやウォレットの補助機能により、受取人ウォレットアドレスから対応するATA（Associated Token Account）を解決し、必要なら作成した上で送金できる。

- ウォレットアドレス指定で送れるように見える理由: ATA自動解決
- `--fund-recipient` の意味: 受取側にATAが無い場合、作成費用を送信側が負担して作成

このため、利用者視点では「ウォレットアドレスだけで送れる」が、内部では最終的にToken Account間で残高移動が行われている。

## 9. A/Bウォレットでの送金検証

### 9.1 Bウォレット作成（デフォルト切り替えなし）

```bash
solana-keygen new --outfile ~/b-wallet.json
solana address -k ~/b-wallet.json
solana balance -k ~/b-wallet.json
```

`--outfile`で作成しただけではCLIのデフォルトキーペアは変わらない。Bとして操作する場合は`-k ~/b-wallet.json`や`--owner ~/b-wallet.json`を明示する。

### 9.2 B側の受取確認

```bash
spl-token accounts --owner ~/b-wallet.json
spl-token balance <MINTアドレス> --owner ~/b-wallet.json
```

AからBへ送金後、BのToken Accountに残高が反映されることを確認できた。
