### 2.3 動作確認とプロジェクト作成

```bash
anchor --version   # anchor-cli 0.32.1

cd mycalculatordapp

```

`anchor init`で雛形プロジェクトが生成される。

## 3. Anchorプロジェクトのフォルダ構造

`mycalculatordapp`プロジェクトの主要なファイルとその役割を整理する。

| パス | 役割 |
|------|------|
| `programs/mycalculatordapp/src/lib.rs` | Rustで書くオンチェーンプログラム本体 |
| `tests/mycalculatordapp.ts` | TypeScriptで書くテストコード |
| `target/idl/mycalculatordapp.json` | プログラムのIDL（命令・引数・アカウントの型情報） |
| `target/types/mycalculatordapp.ts` | IDLから自動生成されるTypeScript型定義 |
| `target/deploy/mycalculatordapp.so` | ビルド済みプログラムバイナリ |
| `Anchor.toml` | クラスタ、プログラムID、テストスクリプト等の設定 |
| `package.json` | Node.js依存パッケージの定義 |
| `jest.config.js` | Jestテストランナーの設定 |
| `tsconfig.json` | TypeScriptコンパイラ設定 |

### 3.1 `programs/` と `tests/` の関係

```
programs/mycalculatordapp/src/lib.rs   ← Rust: オンチェーンロジック（命令の実装）
        ↓ anchor build で IDL 生成
target/idl/mycalculatordapp.json       ← JSON: 命令・型の橋渡し情報
target/types/mycalculatordapp.ts       ← TS型: IDLから自動生成
        ↓ テストから import
tests/mycalculatordapp.ts              ← TS: テスト（命令呼び出し＋検証）
```

この流れがAnchor開発の基本である。Rust側で命令を定義し、`anchor build`でIDLが生成され、TypeScript側からそのIDLを使って型安全に命令を呼び出す。

### 3.2 `target/` の位置づけ

`target/`はビルド成果物置き場であり、手動で編集するものではない。
ただし、テスト時に重要な`IDL`と`型定義`が入るため、TypeScriptテストからの型安全な呼び出しに直結する。
`anchor build`を実行するたびに`target/idl/`と`target/types/`は再生成される。

### 4.2 1行目: `use anchor_lang::prelude::*;`

```rust
use anchor_lang::prelude::*;
```

Anchorフレームワークの基本的な型やマクロをまとめてインポートする。
これにより`Context`、`Account`、`Signer`、`Program`、`Result`、`#[program]`、`#[account]`、`#[derive(Accounts)]`、`declare_id!`等が使えるようになる。

Rustの`use ... ::*`は「そのモジュール内のpublic要素を全て現在のスコープに取り込む」意味。
`prelude`（プレリュード）は「よく使う型をまとめた便利モジュール」の慣例的な名前である。

### 4.3 3行目: `declare_id!`

```rust
declare_id!("5GVm24BPotKvyU5f4GESEHHignTNxsHpJu2gHJ1xvye7");
```

このプログラムのオンチェーンアドレス（プログラムID）を宣言する。
Solanaではプログラムもアカウントの一種であり、このIDでプログラムを一意に識別する。

このIDは`anchor init`時に`target/deploy/mycalculatordapp-keypair.json`から自動生成される。
`Anchor.toml`の`[programs.localnet]`セクションにも同じIDが記載されており、両者が一致している必要がある。

### 4.4 5〜33行目: `#[program]` ブロック — 命令（Instruction）の定義

```rust
#[program]
pub mod mycalculatordapp {
    use super::*;
    // ... 各命令
}
```

`#[program]`はAnchorのマクロで、このブロック内の各`pub fn`が外部から呼び出し可能な**命令（instruction）**になる。
`use super::*;`は親スコープ（ファイル直下）のインポート内容をこのモジュール内でも使えるようにする。

#### `create` 命令

```rust
pub fn create(ctx: Context<Create>, init_message: String) -> Result<()> {
    let calculator = &mut ctx.accounts.calculator;
    calculator.greeting = init_message;
    Ok(())
}
```

| 要素 | 意味 |
|------|------|
| `ctx: Context<Create>` | この命令で使うアカウント群を`Create`構造体の形で受け取る |
| `init_message: String` | 命令の引数。呼び出し側から渡される文字列 |
| `-> Result<()>` | 成功なら`Ok(())`、失敗なら`Err`を返す。Anchor標準の戻り値型 |
| `&mut ctx.accounts.calculator` | `ctx`（コンテキスト）から`calculator`アカウントを可変参照で取得 |
| `calculator.greeting = init_message` | アカウントの`greeting`フィールドに引数の文字列を保存 |

**重要**: Solanaプログラムでは、計算結果を関数の戻り値で返すのではなく、**アカウントの状態として保存**する。これはSolanaが状態遷移モデルを採用しているためである。

#### `add` / `subtract` / `multiply` 命令

```rust
pub fn add(ctx: Context<Addition>, num1: i64, num2: i64) -> Result<()> {
    let calculator = &mut ctx.accounts.calculator;
    calculator.result = num1 + num2;
    Ok(())
}
```

3つとも同じパターンで、演算子だけが異なる。
`i64`は64ビット符号付き整数型（-9,223,372,036,854,775,808 〜 9,223,372,036,854,775,807）。

| 命令 | 演算 | 更新フィールド |
|------|------|----------------|
| `add` | `num1 + num2` | `result` |
| `subtract` | `num1 - num2` | `result` |
| `multiply` | `num1 * num2` | `result` |

#### `divide` 命令

```rust
pub fn divide(ctx: Context<Division>, num1: i64, num2: i64) -> Result<()> {
    let calculator = &mut ctx.accounts.calculator;
    calculator.result = num1 / num2;
    calculator.reminder = num1 % num2;
    Ok(())
}
```

`divide`だけは2つのフィールドを更新する。

| 演算子 | 意味 | 例（17, 3） | 結果 |
|--------|------|-------------|------|
| `/` | 整数除算（商） | `17 / 3` | `5` |
| `%` | 剰余（余り） | `17 % 3` | `2` |

商は`result`に、余りは`reminder`に保存される。

### 4.5 36〜67行目: `#[derive(Accounts)]` — アカウント制約の定義

各命令は、どのアカウントを使うかを事前に宣言する必要がある。
これを**アカウント制約（Account Constraints）**と呼ぶ。

#### `Create` 構造体（アカウント初期化用）

```rust
#[derive(Accounts)]
pub struct Create<'info> {
    #[account(init, payer=user, space=264)]
    pub calculator: Account<'info, Calculator>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}
```

| 要素 | 意味 |
|------|------|
| `#[derive(Accounts)]` | Anchorマクロ。この構造体が命令のアカウント入力であることを宣言 |
| `'info` | Rustのライフタイムパラメータ。アカウント参照の有効期間を示す |
| `#[account(init, payer=user, space=264)]` | 新規アカウント作成。`payer=user`で作成費用をuserが負担。`space=264`バイト確保 |
| `Account<'info, Calculator>` | `Calculator`型のデータを持つアカウント |
| `#[account(mut)]` | このアカウントを書き込み可能として受け取る（SOL残高が減るため） |
| `Signer<'info>` | トランザクションの署名者。本人確認済みのアカウント |
| `Program<'info, System>` | Solanaのシステムプログラム。アカウント作成に必要 |

`space=264`の内訳:
- 8バイト: Anchorのディスクリミネータ（アカウント種別識別子）
- 4 + 200バイト: `greeting`（String = 長さプレフィックス4バイト + 最大200バイトの文字列データ）
- 8バイト: `result`（i64）
- 8バイト: `reminder`（i64）
- 合計: 228バイト以上が必要（264は余裕を持った値）

#### `Addition` / `Subtraction` / `Multiplication` / `Division` 構造体

```rust
#[derive(Accounts)]
pub struct Addition<'info> {
    #[account(mut)]
    pub calculator: Account<'info, Calculator>
}
```

4つとも同じ構造。既存の`calculator`アカウントを`mut`（書き込み可能）で受け取るだけ。
新規作成ではないため、`init`、`payer`、`system_program`は不要。

### 4.6 69〜74行目: `#[account]` — 状態構造体の定義

```rust
#[account]
pub struct Calculator {
    pub greeting: String,
    pub result: i64,
    pub reminder: i64,
}
```

| フィールド | 型 | 用途 |
|-----------|-----|------|
| `greeting` | `String` | `create`命令で設定される初期メッセージ |
| `result` | `i64` | 四則演算の結果（`add`/`subtract`/`multiply`では商） |
| `reminder` | `i64` | `divide`命令の余り。それ以外の命令では更新されない |

`#[account]`マクロにより、このデータ構造はSolanaアカウントのデータ領域にシリアライズ/デシリアライズされるようになる。
つまり、この構造体がオンチェーンに永続化される「状態」である。

### 4.7 Rust構文の補足（初学者向け）

| 構文 | 意味 |
|------|------|
| `&mut` | 可変参照。データを変更するために必要 |
| `ctx.accounts.calculator` | コンテキストからアカウントにアクセスするドット記法 |
| `Ok(())` | 「成功、返す値なし」を意味するResult型の値 |
| `pub` | パブリック（外部からアクセス可能） |
| `pub fn` | パブリック関数 |
| `pub mod` | パブリックモジュール |
| `pub struct` | パブリック構造体 |

## 5. `mycalculatordapp.ts` のコード詳細解説

対象: `tests/mycalculatordapp.ts`

### 5.1 全体コード

```typescript
import * as anchor from "@coral-xyz/anchor";
import { Program } from "@coral-xyz/anchor";
import { Keypair } from "@solana/web3.js";
import { fromWorkspace, LiteSVMProvider } from "anchor-litesvm";
import { Mycalculatordapp } from "../target/types/mycalculatordapp";
import IDL from "../target/idl/mycalculatordapp.json";

describe("mycalculatordapp", () => {
  let provider: LiteSVMProvider;
  let program: Program<Mycalculatordapp>;
  const calculator = Keypair.generate();

  beforeAll(() => {
    const svm = fromWorkspace("./").withBuiltins().withSysvars();
    provider = new LiteSVMProvider(svm);
    anchor.setProvider(provider);
    program = new Program<Mycalculatordapp>(IDL as any, provider);
  });

  it("Creates a calculator", async () => {
    await program.methods
      .create("welcome to Solana")
      .accounts({
        calculator: calculator.publicKey,
        user: provider.wallet.publicKey,
      })
      .signers([calculator])
      .rpc();

    const account = await program.account.calculator.fetch(
      calculator.publicKey
    );
    expect(account.greeting).toBe("welcome to Solana");
  });

  it("Adds two numbers", async () => {
    await program.methods
      .add(new anchor.BN(2), new anchor.BN(3))
      .accounts({
        calculator: calculator.publicKey,
      })
      .rpc();

    const account = await program.account.calculator.fetch(
      calculator.publicKey
    );
    expect(account.result.eq(new anchor.BN(5))).toBe(true);
  });

  it("Subtracts two numbers", async () => {
    await program.methods
      .subtract(new anchor.BN(10), new anchor.BN(3))
      .accounts({
        calculator: calculator.publicKey,
      })
      .rpc();

    const account = await program.account.calculator.fetch(
      calculator.publicKey
    );
    expect(account.result.eq(new anchor.BN(7))).toBe(true);
  });

  it("Multiplies two numbers", async () => {
    await program.methods
      .multiply(new anchor.BN(4), new anchor.BN(5))
      .accounts({
        calculator: calculator.publicKey,
      })
      .rpc();

    const account = await program.account.calculator.fetch(
      calculator.publicKey
    );
    expect(account.result.eq(new anchor.BN(20))).toBe(true);
  });

  it("Divides two numbers", async () => {
    await program.methods
      .divide(new anchor.BN(17), new anchor.BN(3))
      .accounts({
        calculator: calculator.publicKey,
      })
      .rpc();

    const account = await program.account.calculator.fetch(
      calculator.publicKey
    );
    expect(account.result.eq(new anchor.BN(5))).toBe(true);
    expect(account.reminder.eq(new anchor.BN(2))).toBe(true);
  });
});
```

### 5.4 13〜18行目: `beforeAll` — テスト前の初期化処理

### 5.5 20〜34行目: 「Creates a calculator」テスト

```typescript
it("Creates a calculator", async () => {
  await program.methods
    .create("welcome to Solana")
    .accounts({
      calculator: calculator.publicKey,
      user: provider.wallet.publicKey,
    })
    .signers([calculator])
    .rpc();

  const account = await program.account.calculator.fetch(
    calculator.publicKey
  );
  expect(account.greeting).toBe("welcome to Solana");
});
```

このテストの処理フロー:

```
1. program.methods.create(...)  → Rust側の create 命令を呼び出す
2. .accounts({...})             → 命令に渡すアカウントを指定
3. .signers([calculator])       → calculator キーペアで署名
4. .rpc()                       → トランザクションを送信・実行
5. program.account.calculator.fetch(...)  → アカウントの状態を読み取り
6. expect(...).toBe(...)        → Jest で結果を検証
```

各メソッドの詳細:

| メソッド | 意味 |
|----------|------|
| `.create("welcome to Solana")` | Rust側`create`命令の`init_message`引数に文字列を渡す |
| `.accounts({...})` | Rust側`Create`構造体のフィールドに対応するアカウントを指定 |
| `.signers([calculator])` | 新規作成される`calculator`アカウントの秘密鍵で署名（`init`制約で必須） |
| `.rpc()` | トランザクションをLiteSVM上で実行する |
| `program.account.calculator.fetch(...)` | オンチェーンに保存された`Calculator`構造体のデータを取得 |
| `expect(account.greeting).toBe("welcome to Solana")` | `greeting`フィールドが期待値と一致するか検証 |

**なぜ`.signers([calculator])`が必要か**: Anchorの`init`制約では、新規作成されるアカウントのキーペア（秘密鍵）による署名が求められる。これにより「このアドレスでアカウントを作って良い」という承認が行われる。

**なぜ`user`の署名は不要か**: `provider.wallet`はLiteSVMが自動的に署名してくれるため、明示的に`signers`に含める必要がない。

### 5.6 36〜48行目: 「Adds two numbers」テスト

```typescript
it("Adds two numbers", async () => {
  await program.methods
    .add(new anchor.BN(2), new anchor.BN(3))
    .accounts({
      calculator: calculator.publicKey,
    })
    .rpc();

  const account = await program.account.calculator.fetch(
    calculator.publicKey
  );
  expect(account.result.eq(new anchor.BN(5))).toBe(true);
});
```

`create`との違い:

| 観点 | `create` | `add` |
|------|----------|-------|
| 引数 | 文字列1つ | `BN`（大きな整数）2つ |
| アカウント | `calculator` + `user` | `calculator`のみ |
| `.signers()` | 必要（`init`のため） | 不要（既存アカウントの更新） |
| 検証対象 | `greeting` | `result` |

**なぜ`new anchor.BN(2)`を使うのか**: Rust側は`i64`（64ビット整数）だが、JavaScriptの`number`型は53ビットまでしか安全に扱えない。そのため、Anchorでは整数の受け渡しに`BN`（Big Number）ライブラリを使う。比較も`.eq()`メソッドで行う。

### 5.7 50〜91行目: Subtract / Multiply / Divide テスト

`subtract`と`multiply`は`add`と同じパターンで、演算と期待値が異なるだけ。

`divide`テストのみ、検証が2つある:

```typescript
expect(account.result.eq(new anchor.BN(5))).toBe(true);    // 商
expect(account.reminder.eq(new anchor.BN(2))).toBe(true);  // 余り
```

これはRust側の`divide`命令が`result`と`reminder`の2フィールドを更新するため、両方を検証する。

### 5.8 Rust命令とTSテストの対応一覧

| Rust命令 | TS呼び出し | 入力例 | 検証 |
|----------|-----------|--------|------|
| `create(ctx, init_message)` | `.create("welcome to Solana")` | `"welcome to Solana"` | `greeting == "welcome to Solana"` |
| `add(ctx, num1, num2)` | `.add(BN(2), BN(3))` | `2, 3` | `result == 5` |
| `subtract(ctx, num1, num2)` | `.subtract(BN(10), BN(3))` | `10, 3` | `result == 7` |
| `multiply(ctx, num1, num2)` | `.multiply(BN(4), BN(5))` | `4, 5` | `result == 20` |
| `divide(ctx, num1, num2)` | `.divide(BN(17), BN(3))` | `17, 3` | `result == 5`, `reminder == 2` |

## 6. JestとLiteSVMとは何か

JestとLiteSVMは似たものではなく、役割が全く異なる。

| 要素 | 役割 | 具体的に行うこと |
|------|------|------------------|
| **Jest** | テストランナー | `describe`/`it`を実行し、`expect`で合否判定する |
| **LiteSVM** | 軽量Solana実行環境 | テスト内でプログラム実行、アカウント更新、結果取得を行う |

### 6.1 Jestの位置づけ

Jestは「テストを回す仕組み」であり、Solana固有の処理を持たない。
例: `expect(account.result.eq(new anchor.BN(5))).toBe(true);`のようなアサーション評価はJestの仕事。

Jestが提供する主な機能:

| 機能 | 構文 | 役割 |
|------|------|------|
| テストスイート | `describe("name", () => {...})` | テストをグループ化 |
| テストケース | `it("name", async () => {...})` | 個別のテスト定義 |
| 前処理 | `beforeAll(() => {...})` | 全テスト前に1回実行 |
| アサーション | `expect(value).toBe(expected)` | 値の検証 |

### 6.2 LiteSVMの位置づけ

LiteSVMは「ローカルで高速にSolanaプログラムを実行するためのVM」で、`solana-test-validator`よりも軽量。
プロセス内で実行するため、外部バリデータの起動が不要で、テスト実行が非常に高速（約1〜2秒）。

今回のテストでは、`anchor-litesvm`を使って次を行っている:

1. ワークスペースからビルド済みプログラム（`.so`）を読み込む
2. ProviderをLiteSVMに切り替える
3. RPC通信ではなく、プロセス内で命令実行を検証する

### 6.3 `solana-test-validator`との比較

| 観点 | `solana-test-validator` | LiteSVM |
|------|------------------------|---------|
| 実行方式 | 外部プロセスとして起動 | テストプロセス内で実行 |
| 起動時間 | 数秒〜十数秒 | ほぼ瞬時 |
| テスト速度 | RPC通信のオーバーヘッドあり | プロセス内で直接実行 |
| 時間操作 | 困難 | `setClock()`で自由に操作可能 |
| 本番との近さ | より本番に近い動作 | 軽量だが一部挙動が異なる可能性 |

### 6.4 使い分けの結論

- **Jest**: テストの「進行役（実行と判定）」
- **LiteSVM**: Solanaの「実行環境（チェーン代替）」

つまり、`Jest + LiteSVM`は「ランナー + 実行環境」の組み合わせである。

## 8. 実行時に遭遇したエラーと理解

今回、テスト過程で次の典型エラーが発生した。

| エラー | 原因 | 対処 |
|--------|------|------|
| `command not found: anchor` | Anchor CLI未導入またはPATH未反映 | `source ~/.zshrc`でPATH反映 |
| `Your configured rpc port: 8899 is already in use` | 既存`solana-test-validator`とのポート競合 | 既存プロセスを停止してから再実行 |
| `Cannot find module '@project-serum/anchor'` | 旧パッケージ名を使用 | `@coral-xyz/anchor`に変更 |
| `Program is not deployed / Unsupported program id` | バリデータ状態とデプロイ状態の不整合 | `anchor build`でリビルド、またはSolana CLI更新 |
| `ProgramResult<()>` → コンパイルエラー | Anchorでは`Result<()>`が正しい戻り値型 | `ProgramResult<()>`を`Result<()>`に修正 |

これらは、Solana/Anchor初学者がほぼ確実に通る論点であり、今回の学習で切り分け手順を経験できた。
