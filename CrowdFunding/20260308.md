**日付**: 2026年3月8日
**学習内容**: Anchorによるクラウドファンディングプログラムの実装、PDA（Program Derived Address）によるアカウント生成、SOLの送金（CPI）と直接lamports操作の違い、`system_program`の要否の理解

## 1. 学習の全体像

本日の学習では、前回の電卓プログラムから一歩進み、実用的な**クラウドファンディングプログラム**をAnchorで実装した。
特に、次の3点を重点的に理解した。

- キャンペーン（Campaign）の作成・寄付・引き出しという3つの命令の実装
- PDA（Program Derived Address）を使ったアカウントアドレスの自動導出
- SOLを動かす2つの方法（CPI経由 vs lamports直接操作）と`system_program`が必要になる条件

## 2. プロジェクトのセットアップ

### 2.1 Anchorプロジェクトの初期化

```bash
anchor init crowdfunding
cd crowdfunding
anchor build
```

`anchor init crowdfunding`で新しいAnchorプロジェクトが生成される。
前回の電卓プロジェクトと同じ手順だが、今回のプログラム名は`crowdfunding`になる。

### 2.2 Anchorバージョンの確認

`programs/crowdfunding/Cargo.toml`に以下の記述がある。

```toml
[dependencies]
anchor-lang = "0.32.1"
```

この行でAnchorフレームワークのバージョンを指定する。プロジェクト内のRustコードは、このバージョンのAnchor APIに基づいて書く。

### 2.3 IDL（Interface Definition Language）の自動生成

`anchor build`を実行すると、`target/idl/crowdfunding.json`にプログラムのIDLが自動生成される。

IDLとは「プログラムの取扱説明書」のようなもので、以下の情報がJSON形式で記述される。

- どんな命令（instruction）があるか
- 各命令にどんな引数が必要か
- 各命令にどんなアカウントを渡すか
- アカウントのデータ構造（型情報）

フロントエンドやテストコードは、このIDLを読み込むことで、プログラムの命令を型安全に呼び出せるようになる。

### 2.4 IDL（JSON）とTypeScript型定義（TS）の違い

`anchor build`を実行すると、`crowdfunding.json`と`crowdfunding.ts`という似た2つのファイルが生成される。これらは**役割が異なる**。

```
crowdfunding.json（IDL）  = 「データ」 → 実行時に読み込まれる
crowdfunding.ts（型定義） = 「型情報」 → 開発時にだけ使われる
```

**`crowdfunding.json`（IDL） — 実行時に必要**:

プログラムが実際に動くときに使う。TypeScriptのコードがこのJSONを読み込んで、「`create`命令のデータはこう組み立てる」「引数はこの順番でバイト列に変換する」といった処理を行う。

```typescript
// テストやフロントエンドで、JSONを読み込んでプログラムクライアントを作る
import IDL from "../target/idl/crowdfunding.json";
const program = new Program(IDL, provider);  // ← 実行時にJSONの中身を使う
```

JSONがなければ、プログラムの命令を呼び出すことすらできない。

**`crowdfunding.ts`（型定義） — 開発時に必要**:

コードを書いているときにだけ使う。VSCodeの自動補完やエラー表示のためのもの。実行時には削除しても動く。

```typescript
// 型定義をimportすると、VSCodeが補完・エラー表示してくれる
import { Crowdfunding } from "../target/types/crowdfunding";
const program = new Program<Crowdfunding>(IDL, provider);
//                          ^^^^^^^^^^^^ これが型定義
```

型定義があると、間違った引数を渡したときに**コードを書いた段階で**エラーに気づける。

```typescript
// 型定義ありの場合
program.methods.create("Campaign", "Desc")  // OK（string, string）
program.methods.create(123, true)            // 赤線でエラー表示！

// 型定義なしの場合
program.methods.create(123, true)            // コンパイルは通るが、実行時にエラー
```

**比較表**:

| 観点 | `crowdfunding.json`（IDL） | `crowdfunding.ts`（型定義） |
|------|---------------------------|---------------------------|
| 使われるタイミング | **実行時**（プログラムを呼ぶとき） | **開発時**（コードを書くとき） |
| なくなったら | プログラムを呼べない（動かない） | 動くが、補完やエラー表示が効かない |
| 中身 | 命令・引数・アカウントの構造（データ） | 同じ情報のTypeScript型（型チェック用） |
| 読む相手 | JavaScriptエンジン（Node.js等） | TypeScriptコンパイラ / VSCode |

例えるなら、JSONは「レストランのメニュー表（注文するときに実際に必要）」、TS型は「メニューの写真付きガイド（選ぶときに便利だが、なくても注文はできる）」である。

## 4. `lib.rs` のコード詳細解説

対象: `programs/crowdfunding/src/lib.rs`

### 4.1 全体コード

```rust
use anchor_lang::prelude::*;

declare_id!("FiPBwdr9QMkpeJuXhm5Dr2tzxN8DbYod2g6Q9eEKvYdL");

#[program]
pub mod crowdfunding {
    use super::*;

    pub fn create(ctx: Context<Create>, name: String, description: String) -> Result<()> {
        let campaign = &mut ctx.accounts.campaign;
        campaign.name = name;
        campaign.description = description;
        campaign.amount_donated = 0;
        campaign.admin = ctx.accounts.user.key();
        Ok(())
    }

    pub fn withdraw(ctx: Context<Withdraw>, amount: u64) -> Result<()> {
        let campaign = &mut ctx.accounts.campaign;
        let user = &mut ctx.accounts.user;
        if campaign.admin != user.key() {
            return Err(ProgramError::InvalidInstructionData.into());
        }
        let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
        if campaign.to_account_info().lamports() - rent_balance < amount {
            return Err(ProgramError::InsufficientFunds.into());
        }
        let campaign_info = campaign.to_account_info();
        let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
        **campaign_lamports = (*campaign_lamports).saturating_sub(amount);
        let user_info = user.to_account_info();
        let mut user_lamports = user_info.try_borrow_mut_lamports()?;
        **user_lamports = (*user_lamports).saturating_add(amount);
        Ok(())
    }

    pub fn donate(ctx: Context<Donate>, amount: u64) -> Result<()> {
        let ix = anchor_lang::solana_program::system_instruction::transfer(
            &ctx.accounts.user.key(),
            &ctx.accounts.campaign.key(),
            amount
        );
        anchor_lang::solana_program::program::invoke(
            &ix,
            &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
        )?;
        (&mut ctx.accounts.campaign).amount_donated += amount;
        Ok(())
    }
}

#[derive(Accounts)]
pub struct Create<'info> {
    #[account(init, payer = user, space = 9000, seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()], bump)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}

#[derive(Accounts)]
pub struct Withdraw<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
}

#[derive(Accounts)]
pub struct Donate<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}

#[account]
pub struct Campaign {
    pub admin: Pubkey,
    pub name: String,
    pub description: String,
    pub amount_donated: u64
}
```

#### 同じアドレスにデプロイすると上書きされるのか？

**同じアドレスにデプロイすると、既存のプログラムは上書きされる。** ただし、いくつか重要な条件がある。

**上書きできる条件**:

| 条件 | 説明 |
|------|------|
| アップグレード権限を持っている | デプロイ時に使ったウォレット（Upgrade Authority）と同じ秘密鍵でないと上書きできない |
| プログラムがアップグレード可能な状態 | デフォルトでは`upgradeable`（上書き可能）でデプロイされる |

つまり、自分の秘密鍵（`~/.config/solana/id.json`）を持っている人だけが上書きできる。他人が同じアドレスに勝手にデプロイすることはできない。

**ネットワークごとの独立性**:

```
localnet  → FiPBwdr9Q... のプログラム（ローカルだけに存在）
devnet    → FiPBwdr9Q... のプログラム（devnetだけに存在）
mainnet   → FiPBwdr9Q... のプログラム（mainnetだけに存在）
```

各ネットワークは完全に独立しており、localnetにデプロイしてもdevnetやmainnetには影響しない。

**上書きを禁止する方法**:

本番運用時に「もう二度とコードを変更しない」と決めた場合、以下のコマンドでアップグレード権限を放棄できる。

```bash
solana program set-upgrade-authority <PROGRAM_ID> --final
```

これを実行すると、誰もプログラムを上書きできなくなる（不可逆）。

**Ethereumとの比較**:

| 観点 | Ethereum | Solana |
|------|----------|--------|
| デフォルト | **イミュータブル**（上書き不可） | **アップグレード可能**（上書き可能） |
| 上書きしたい場合 | Proxyパターン等の工夫が必要 | そのまま`solana program deploy`するだけ |
| 上書き禁止 | デフォルトでそうなっている | `--final`フラグで明示的に設定 |

Solanaの方が「開発中は気軽にバグ修正できる」設計になっている。

**ユーザー側からプログラムの状態を確認する方法**:

誰でも以下のコマンドでプログラムの状態を確認できる。

```bash
solana program show <PROGRAM_ID>
```

出力例:

```
Program Id: FiPBwdr9QMkpeJuXhm5Dr2tzxN8DbYod2g6Q9eEKvYdL
Owner: BPFLoaderUpgradeab1e11111111111111111111111
ProgramData Address: 3Vg9...xxxx
Authority: 7nYB...yyyy        ← Upgrade Authorityの公開鍵
Last Deployed In Slot: 12345
Data Length: 200000 bytes
```

| フィールド | 意味 |
|-----------|------|
| `Authority` | アップグレード権限を持つウォレットの公開鍵。**ここが表示されていれば上書き可能** |
| `Authority: None` | アップグレード権限が放棄済み。**誰も上書きできない（イミュータブル）** |

ユーザーは`Authority`の有無を見るだけで「このプログラムは今後変更される可能性があるか」を判断できる。DeFiプロジェクトなどでは、`Authority: None`であることが信頼性の指標として重視される。開発者がいつでもコードを書き換えられる状態は、ユーザーにとってリスクになるためである。

### 4.4 5〜7行目: `#[program]` ブロックと `use super::*;`

```rust
#[program]
pub mod crowdfunding {
    use super::*;
    // ... 各命令
}
```

`#[program]`はAnchorのマクロで、このブロック内の各`pub fn`が外部から呼び出し可能な命令（instruction）になる。

**`pub mod crowdfunding` の意味**:

| 部分 | 意味 |
|------|------|
| `pub` | 外部からアクセス可能にする |
| `mod` | モジュール（コードをグループ化する箱）を作る |
| `crowdfunding` | モジュールの名前 |

Rustの`mod`は「関連するコードをまとめるフォルダ」のようなもの。この中に`create`、`withdraw`、`donate`という3つの命令をまとめている。

**`use super::*;` の意味**:

`mod`の中は独立した名前空間になるため、外側（1行目）で`use anchor_lang::prelude::*;`でインポートした型（`Context`、`Result`など）がそのままでは使えない。

```
ファイル全体のスコープ（親）
├── use anchor_lang::prelude::*;   ← ここでインポート
├── declare_id!(...);
│
└── mod crowdfunding（子）          ← 別の名前空間
    └── use super::*;              ← 「親のインポートを全部持ってきて」
        ├── create(...)            ← Context, Result 等が使える
        ├── withdraw(...)
        └── donate(...)
```

| 部分 | 意味 |
|------|------|
| `use` | 他の場所で定義されたものを使えるようにする |
| `super` | 「一つ上の階層（親モジュール）」を指す |
| `::*` | 「その中のpublic要素を全部」 |

つまり`use super::*;`は「親（ファイル直下）でインポート済みの`Context`や`Result`などを、このモジュール内でも使えるようにする」という意味。もし`use super::*;`を書き忘れると、`Context`や`Result`が見つからずコンパイルエラーになる。

**`use anchor_lang::prelude::*;`（1行目）との関係**:

この2つの`use`文は連携して動いている。

| `use`文 | 場所 | 役割 |
|---------|------|------|
| `use anchor_lang::prelude::*;` | ファイル直下（親） | Anchorの基本型（`Context`、`Account`、`Signer`、`Result`等）をファイル全体で使えるようにする |
| `use super::*;` | `mod crowdfunding`の中（子） | 親でインポート済みの型を、このモジュール内にも引き継ぐ |

`prelude`（プレリュード）は「よく使う型をまとめた便利モジュール」の慣例的な名前で、Anchorでは以下の型が含まれる。

| 型 | 用途 |
|----|------|
| `Context` | 命令のアカウント群を受け取る型 |
| `Account` | オンチェーンアカウントを表す型 |
| `Signer` | トランザクション署名者を表す型 |
| `Program` | 外部プログラムを表す型 |
| `System` | System Programを表す型 |
| `Result` | 成功/失敗を返す型 |
| `Pubkey` | 公開鍵を表す型 |
| `Rent` | Rent情報を取得する型 |
| `ProgramError` | エラー型 |

これらを1行でまとめてインポートできるのが`use anchor_lang::prelude::*;`の役割である。

このプログラムには3つの命令がある。

| 命令 | 役割 | 引数 |
|------|------|------|
| `create` | キャンペーンを新規作成する | `name`（名前）、`description`（説明） |
| `withdraw` | キャンペーンからSOLを引き出す | `amount`（引き出し額） |
| `donate` | キャンペーンにSOLを寄付する | `amount`（寄付額） |

#### 4.4.1 `create` 命令 — キャンペーンの作成

```rust
pub fn create(ctx: Context<Create>, name: String, description: String) -> Result<()> {
    let campaign = &mut ctx.accounts.campaign;
    campaign.name = name;
    campaign.description = description;
    campaign.amount_donated = 0;
    campaign.admin = ctx.accounts.user.key();
    Ok(())
}
```

この命令は新しいクラウドファンディングのキャンペーンを作成する。

**注意: `create`はデプロイ時に自動実行されない**:

Ethereumでは`constructor`がデプロイ時に1回だけ自動実行され、初期化処理を行う。しかしSolanaにはその仕組みがない。全ての命令は外部から明示的にトランザクションを送って呼び出す必要がある。

| ステップ | 何が起こるか |
|---------|-------------|
| `anchor build` | Rustコードをバイナリ（`.so`）にコンパイルする |
| `anchor deploy` | バイナリをSolanaネットワークにアップロードする。**プログラムが使える状態になるだけ** |
| `create`命令の呼び出し | ユーザー（またはテスト）が**明示的にトランザクションを送って**初めてキャンペーンが作成される |

つまりデプロイ直後のプログラムは「命令を受け付ける準備ができた状態」であり、キャンペーンなどのデータはまだ存在しない。

| 行 | 処理内容 |
|----|----------|
| `let campaign = &mut ctx.accounts.campaign;` | コンテキストから`campaign`アカウントを可変参照（書き込み可能な形）で取得する（後述） |
| `campaign.name = name;` | 引数で受け取ったキャンペーン名を保存する |
| `campaign.description = description;` | 引数で受け取ったキャンペーン説明を保存する |
| `campaign.amount_donated = 0;` | 寄付総額を0で初期化する |
| `campaign.admin = ctx.accounts.user.key();` | キャンペーンを作った人（署名者）のアドレスを管理者として記録する |
| `Ok(())` | 「処理が成功しました」を返す |

**ポイント**: `ctx.accounts.user.key()`でトランザクションの署名者（=キャンペーン作成者）の公開鍵を取得し、`admin`フィールドに保存している。これにより、後の`withdraw`命令で「管理者本人だけが引き出しできる」という権限チェックが可能になる。

**`let campaign = &mut ctx.accounts.campaign;` の詳細解説**:

この1行を理解するには、まずSolanaの「コードとデータの分離」という設計思想を知る必要がある。

**前提: Solanaではコードとデータは別々のアカウントに保存される**:

従来のプログラム（PythonやJavaScript等）では、データは変数としてプログラムの中に存在する。しかしSolanaでは、プログラム（コード）とデータ（アカウント）は完全に分離されている。

```
従来のプログラム:
  データは変数としてプログラムの中にある
  name = "Campaign"   ← プログラムの中にデータがある

Solana:
  プログラム → コードだけ。データを持てない
  アカウント → データだけ。コードを持てない
```

そのため、関数を実行するとき「どのアカウント（データ）を使うか」を外から渡す必要がある。それが`ctx`（Context）である。

**データはどこに保存されるのか**:

データはデプロイした人のウォレットではなく、**`create`命令の実行時に新しく作られるアカウント**に保存される。

```
デプロイした人のウォレット（例: 7nYB...yyyy）
  → SOLを持っているだけ。キャンペーンのデータは入らない

プログラム（例: FiPB...YdL）
  → コードだけ。データは入らない

キャンペーンアカウント（PDAで自動生成されたアドレス）  ← ここにデータが入る！
  ├── admin: 7nYB...yyyy（作った人のアドレス）
  ├── name: "Help Fund Project"
  ├── description: "A great project"
  └── amount_donated: 0
```

| アカウント | 中身 | データを保持するか |
|-----------|------|-------------------|
| デプロイした人のウォレット | SOL残高 | しない |
| プログラム | コード（命令の処理ロジック） | しない |
| キャンペーンアカウント（PDA） | Campaign構造体のデータ | **する** |

つまり`create`命令は「ブロックチェーン上に新しいアカウント（=データの入れ物）を作り、そこにデータを書き込む」処理である。アカウント作成の費用は`payer = user`の指定により、命令を呼び出したユーザーが支払う。

**`ctx` の中身を具体的に見る**:

`Context<Create>`の`Create`は、下の方で定義されている構造体を指す。

```rust
pub struct Create<'info> {
    pub campaign: Account<'info, Campaign>,  // キャンペーンのアカウント
    pub user: Signer<'info>,                 // ユーザーのアカウント
    pub system_program: Program<'info, System> // System Program
}
```

つまり`ctx`の中には、この3つのアカウントが入っている。

```
ctx（箱）
├── ctx.accounts.campaign       ← キャンペーンのデータ
├── ctx.accounts.user           ← ユーザー（署名者）
└── ctx.accounts.system_program ← System Program
```

**`&mut` とは何か**:

`&mut`は「中身を書き換えてもいいですよ」という許可証である。Rustでは変数はデフォルトで**読み取り専用**。データを変更するには`&mut`（可変参照）が必要になる。

日常生活に例えると以下のようになる。

| 操作 | 例え |
|------|------|
| `&`（参照） | 友達のノートを「見せて」と借りる。**読むだけ**。書き込みは禁止 |
| `&mut`（可変参照） | 友達のノートを「ちょっと書き込んでいい？」と借りる。**読むことも書き込むこともできる** |

コードで見ると以下の通り。

```rust
// & だけの場合（読み取り専用）
let campaign = &ctx.accounts.campaign;
let x = campaign.name;        // OK: 読める
campaign.name = "New";         // NG: 書き込めない！エラー！

// &mut の場合（読み書き両方OK）
let campaign = &mut ctx.accounts.campaign;
let x = campaign.name;        // OK: 読める
campaign.name = "New";         // OK: 書き込める！
```

**1行のまとめ**:

```rust
let campaign = &mut ctx.accounts.campaign;
```

| ステップ | 部分 | 意味 |
|---------|------|------|
| 1 | `ctx` | 関数に渡されたアカウント群の箱を開けて |
| 2 | `.accounts.campaign` | その中の`campaign`アカウント（ブロックチェーン上の別アカウント）を取り出して |
| 3 | `&mut` | 「書き込みOK」の許可証つきで |
| 4 | `let campaign =` | `campaign`という名前で使えるようにする |

これにより次の行で`campaign.name = name;`とブロックチェーン上のアカウントにデータを書き込めるようになる。

#### 4.4.2 `withdraw` 命令 — SOLの引き出し

```rust
pub fn withdraw(ctx: Context<Withdraw>, amount: u64) -> Result<()> {
    let campaign = &mut ctx.accounts.campaign;
    let user = &mut ctx.accounts.user;
    // ① 権限チェック: 管理者本人か？
    if campaign.admin != user.key() {
        return Err(ProgramError::InvalidInstructionData.into());
    }
    // ② 残高チェック: 引き出し可能な額があるか？
    let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
    if campaign.to_account_info().lamports() - rent_balance < amount {
        return Err(ProgramError::InsufficientFunds.into());
    }
    // ③ lamportsを直接操作してSOLを移動する
    let campaign_info = campaign.to_account_info();
    let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
    **campaign_lamports = (*campaign_lamports).saturating_sub(amount);
    let user_info = user.to_account_info();
    let mut user_lamports = user_info.try_borrow_mut_lamports()?;
    **user_lamports = (*user_lamports).saturating_add(amount);
    Ok(())
}
```

この命令はキャンペーンに集まったSOLを管理者が引き出すためのもので、3つのステップで構成される。

**ステップ①: 権限チェック**

```rust
if campaign.admin != user.key() {
    return Err(ProgramError::InvalidInstructionData.into());
}
```

キャンペーンの`admin`（作成時に記録された管理者アドレス）と、今回の呼び出し者（`user`）が一致するか確認する。一致しなければエラーを返して処理を中断する。これにより**管理者以外の人はSOLを引き出せない**。

**`return Err(ProgramError::InvalidInstructionData.into());` の詳細解説**:

この1行はエラーを返して関数を即座に終了させる処理である。各パーツの意味を分解する。

| パーツ | 意味 |
|--------|------|
| `return` | 関数を**今すぐ終了**する。この行以降のコードは一切実行されない |
| `Err(...)` | `Result`型の「失敗」側の値を作る。`Ok(())`が「成功」なら、`Err(...)`は「失敗」 |
| `ProgramError::InvalidInstructionData` | Solanaが標準で用意しているエラーの一つ。「命令のデータが不正です」という意味 |
| `.into()` | 型を自動変換する（後述） |

**`Result<()>` と `Ok` / `Err` の関係**:

Rustでは関数の戻り値に`Result<()>`を使うことで、「成功」か「失敗」のどちらかを返す。

```
Result<()>（結果型）
├── Ok(())   ← 成功した場合。()は「特に返すデータなし」の意味
└── Err(e)   ← 失敗した場合。eにはエラーの詳細が入る
```

日常生活に例えると以下の通り。

| Rustのコード | 日常の例え |
|-------------|-----------|
| `Ok(())` | 「作業完了しました！」と報告する |
| `Err(ProgramError::...)` | 「問題が発生しました。理由は〇〇です」と報告する |
| `return Err(...)` | 「問題があるので作業を中止します」と言ってその場を離れる |

**`.into()` とは何か — 型の自動変換**:

`.into()`は「ある型を別の型に変換する」メソッドである。ここでは`ProgramError`（Solana標準のエラー型）を、Anchorフレームワークが使う`Error`型に変換している。

```
ProgramError::InvalidInstructionData   ← Solana標準のエラー型
    │
    │  .into()  ← 型を変換
    ▼
anchor_lang::error::Error              ← Anchorのエラー型
```

なぜ変換が必要かというと、関数の戻り値が`Result<()>`（Anchorの`Result`）であるため、Solana標準の`ProgramError`をそのまま返すと型が合わずコンパイルエラーになるからである。

日常生活に例えると以下のようになる。

| 場面 | 例え |
|------|------|
| `.into()`なし | 日本の100円玉をアメリカの自動販売機に入れようとする → 使えない |
| `.into()`あり | 日本の100円玉を両替所でドルに換えてから自動販売機に入れる → 使える |

`.into()`は「両替」のようなもので、中身（エラーの意味）は変わらないが、形式（型）を相手が受け取れるものに変換する。

**ステップ②: 残高チェック（Rent免除の考慮）**

```rust
let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
if campaign.to_account_info().lamports() - rent_balance < amount {
    return Err(ProgramError::InsufficientFunds.into());
}
```

Solanaではアカウント（データの入れ物）をブロックチェーン上に存在させ続けるために、**Rent（家賃）**という仕組みがある。アパートの家賃と同じで、「場所を占有するならお金を払ってね」という考え方である。

```
アパートの例え:
  部屋が大きい（データが多い） → 家賃が高い
  部屋が小さい（データが少ない） → 家賃が安い
  家賃を払えない → 退去させられる（アカウント削除）
```

ただし、Solanaでは**2年分の家賃を前払い**すると、それ以降は家賃免除（Rent Exempt）になる。現在のSolanaでは実質的にすべてのアカウントがこの「家賃免除」を求められる。

**`Rent::get()?` の各パーツの意味**:

まず`::` は「〜の中の」という意味。`Rent::get()` は「`Rent`という型の中にある`get()`という関数を呼ぶ」ということ。

| Rust | 日常の例え |
|------|-----------|
| `Rent::get()` | 「市役所の窓口で情報をもらう」 |
| `ProgramError::InsufficientFunds` | 「エラー集の中の"残高不足"のページを開く」 |

Pythonでいう`Rent.get()`、JavaScriptでいう`Rent.get()`と同じ概念だが、Rustでは`.`ではなく`::`を使う（インスタンスメソッドは`.`、型に紐づく関数は`::`で呼び出す）。

次に`?`は「もし失敗したら、このエラーをそのまま返して関数を終了する」という省略記法。

```rust
// ?を使う場合（短い）
let rent_balance = Rent::get()?.minimum_balance(data_len);

// ?を使わない場合（長い。やっていることは同じ）
let rent = Rent::get();
let rent = match rent {
    Ok(value) => value,      // 成功 → 値を取り出す
    Err(e) => return Err(e), // 失敗 → エラーを返して関数終了
};
let rent_balance = rent.minimum_balance(data_len);
```

つまり`?`は「成功したら中身を取り出し、失敗したらエラーを返す」を1文字で書ける便利な記号である。

**「現在のRentパラメータをチェーンから取得する」とは**:

`Rent::get()?` は「今のSolanaネットワークでの家賃の料金表を取得する」という意味。この料金表はネットワークの設定値としてチェーン上に存在していて、時期によって変わる可能性がある（実際にはほぼ固定だが、仕組み上は変更可能）。

| パーツ | 意味 |
|--------|------|
| `Rent::get()?` | チェーンから家賃の料金表を取得する |
| `campaign.to_account_info().data_len()` | このキャンペーンアカウントのデータサイズ（バイト数）を取得する |
| `.minimum_balance(...)` | そのデータサイズの部屋を維持するのに最低限必要なSOL（lamports）を計算する |
| `campaign.to_account_info().lamports()` | キャンペーンアカウントの現在のSOL残高（lamports単位）を取得する |

**`campaign.to_account_info().lamports()` の詳細解説**:

この式は2段階の処理で構成されている。

まず`.to_account_info()`は、Anchorの便利な型（`Account<Campaign>`）をSolanaランタイムが管理する**生のアカウント情報**（`AccountInfo`）に変換する。

```
Account<Campaign>（Anchorの便利な型）
  → name, description, amount_donated などのフィールドに簡単にアクセスできる

AccountInfo（Solanaの生の型）
  → lamports（SOL残高）、data（生のバイト列）、owner（所有プログラム）など
    アカウントの「裏側」の情報にアクセスできる
```

日常で例えると以下の通り。

| 型 | 例え |
|---|------|
| `Account<Campaign>` | 商品のパッケージ。ラベル（name, description）が見やすい |
| `AccountInfo` | パッケージを開けた中身。重さ（lamports）や原材料（data）が分かる |

次に`.lamports()`はそのアカウントが持っているSOLの量をlamports単位で返す。`lamports`はSOLの最小単位で、1 SOL = 1,000,000,000 lamports（10億lamports）である。

```
例: キャンペーンに5 SOL集まっている場合
  campaign.to_account_info().lamports()
  → 5,000,000,000（50億lamports）
```

**なぜ`campaign.lamports()`と直接書けないのか**:

`campaign`（`Account<Campaign>`型）は`name`や`description`など**プログラムが定義したデータ**にアクセスするための型。SOL残高（lamports）はプログラムが定義したものではなく、**Solanaランタイムが管理するアカウントの基本情報**なので、`.to_account_info()`で生の情報に変換してからアクセスする必要がある。

```
campaign.name              → OK（Anchorの型で直接アクセスできる）
campaign.description       → OK
campaign.lamports()        → NG（Anchorの型にはlamports()がない）
campaign.to_account_info().lamports()  → OK（生の型に変換してからアクセス）
```

つまりこの行は「このアカウントが削除されないために最低限キープしておくべきSOL額」を計算している。withdrawで全額引き出してしまうとアカウントが消えてしまうので、この額は引き出せないようにガードしている。

「現在の残高 - Rent免除に必要な額」が引き出し額より少なければ、資金不足エラー（`ProgramError::InsufficientFunds`）を返す。

**ステップ③: lamportsの直接操作**

```rust
let campaign_info = campaign.to_account_info();
let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
**campaign_lamports = (*campaign_lamports).saturating_sub(amount);
let user_info = user.to_account_info();
let mut user_lamports = user_info.try_borrow_mut_lamports()?;
**user_lamports = (*user_lamports).saturating_add(amount);
```

ここがSOLの実際の移動処理。SOLの送金というと難しく聞こえるが、やっていることは「キャンペーンの残高を減らし、ユーザーの残高を増やす」だけである。銀行振込のように「Aの口座から引いて、Bの口座に足す」のと同じ。

```
やっていることのイメージ:
  キャンペーンの残高: 10 SOL → 7 SOL（3 SOL減らす）
  ユーザーの残高:      2 SOL → 5 SOL（3 SOL増やす）
```

以下、1行ずつ詳しく解説する。

**1行目: `let campaign_info = campaign.to_account_info();`**

ステップ②で解説した通り、Anchorの便利な型（`Account<Campaign>`）からSolanaの生のアカウント情報（`AccountInfo`）に変換する。SOL残高を操作するには生のアカウント情報が必要なため。

**2行目: `let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;`**

| パーツ | 意味 |
|--------|------|
| `try_borrow_mut_lamports()` | lamports（SOL残高）を「書き換え可能な形」で借りる |
| `try_` | 「失敗する可能性がある」という意味の接頭辞。既に他の場所で借りられていたら失敗する |
| `borrow_mut` | 「書き換え可能な形で借りる」。Rustの所有権ルールにより、同時に2箇所から書き換えることを防ぐ |
| `?` | 失敗したらエラーを返す（ステップ②で解説済み） |
| `let mut` | 変数自体も書き換え可能にする |

日常で例えると「金庫の鍵を借りて、中のお金を動かせる状態にする」こと。`try`がつくのは「他の人が既に鍵を借りていたら失敗する」から。

**3行目: `**campaign_lamports = (*campaign_lamports).saturating_sub(amount);`**

この行はキャンペーンの残高を減らす処理。3つのパーツに分けて解説する。

**(a) `**`（二重デリファレンス）とは何か**:

`try_borrow_mut_lamports()`が返す値は「参照の参照」（箱の中に箱が入っている状態）。実際の数値にたどり着くには`*`を2回使って箱を2つ開ける必要がある。

```
campaign_lamports の中身（2重の箱）:

┌─────────────────────────────────┐
│ 外側の箱（1つ目の参照）           │
│  ┌───────────────────────────┐  │
│  │ 内側の箱（2つ目の参照）     │  │
│  │  ┌─────────────────────┐  │  │
│  │  │ 実際の数値: 10000000 │  │  │
│  │  └─────────────────────┘  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘

*campaign_lamports   → 外側の箱を開ける → まだ箱がある
**campaign_lamports  → 内側の箱も開ける → 数値が見える！
```

**なぜ箱が2重になっているのか**:

まず前提として、Solanaでは1つのトランザクションで**複数のアカウント**を同時に扱う。例えばwithdrawでは`campaign`と`user`の2つのアカウントのlamportsを操作する。もし同じアカウントのlamportsを**2箇所から同時に書き換え**てしまうと、データが壊れる可能性がある。

```
危険な例（もし自由に書き換えられたら）:
  残高: 100

  処理A: 残高を読む → 100     処理B: 残高を読む → 100
  処理A: 100 - 30 = 70 で書く  処理B: 100 - 50 = 50 で書く

  結果: 残高が 50 になる（本来は 100 - 30 - 50 = 20 のはず！）
```

これを防ぐために、Solanaは**「貸出ノート」方式**を使っている。

**1重目の箱: `RefMut`（貸出ノート）**

図書館の本の貸出と同じ仕組み。

```
図書館の例え:
  本（= lamportsの値）を借りるとき
  → 貸出カードに「○○さんが借りています」と記録する
  → 別の人が同じ本を借りようとすると「貸出中です」と断られる
```

`try_borrow_mut_lamports()?`は「lamportsの値を借りる + 貸出カードを発行する」処理。`RefMut`がこの貸出カードに当たる。もし既に誰かが借りていたら、`try_`の部分でエラーになる。

**2重目の箱: `&mut u64`（実際の数値への参照）**

`RefMut`の中には、lamportsの実際の数値（例: 10,000,000,000）への参照（`&mut u64`）が入っている。

**まとめ**:

```
campaign_lamports の構造:

┌─ RefMut（1重目）= 貸出カード ─────────┐
│  「このlamportsは今borrowされています」  │
│                                        │
│  ┌─ &mut u64（2重目）= 本の中身 ────┐  │
│  │  実際の数値: 10,000,000,000      │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘

*campaign_lamports   → 貸出カードを開ける → 本が出てくる
**campaign_lamports  → 本も開ける → 数値が見える！
```

| 箱 | 正体 | 役割 |
|----|------|------|
| 1重目 | `RefMut` | 「今借りています」という安全管理。同時に2箇所から書き換えるのを防ぐ |
| 2重目 | `&mut u64` | lamportsの実際の数値（メモリ上の場所）を指し示す |

つまり2重になっている理由は「安全のための管理層（`RefMut`）」と「実際のデータへのアクセス層（`&mut u64`）」の2つが必要だからである。

**(b) `.saturating_sub(amount)` とは何か**:

「安全な引き算」。普通の引き算（`-`）だと、結果がマイナスになった場合にプログラムがクラッシュする。`saturating_sub`は結果がマイナスになりそうな場合、0で止まる。

```
普通の引き算:  3 - 5 = -2  → パニック！プログラムが停止！
saturating_sub: 3 - 5 = 0   → 0で安全に止まる
```

| メソッド | 意味 | 溢れた場合 |
|---------|------|-----------|
| `saturating_sub(amount)` | 安全な引き算 | 0で止まる |
| `saturating_add(amount)` | 安全な足し算 | u64の最大値で止まる |

**(c) 行全体の意味**:

```rust
**campaign_lamports = (*campaign_lamports).saturating_sub(amount);
```

| ステップ | 部分 | 意味 |
|---------|------|------|
| 1 | `(*campaign_lamports)` | 外側の箱を1つ開けて内側の参照を取り出す |
| 2 | `.saturating_sub(amount)` | その値から`amount`を安全に引き算する |
| 3 | `**campaign_lamports =` | 2重の箱を開けて、実際の数値を計算結果で上書きする |

**4〜6行目: ユーザー側も同じ処理**

```rust
let user_info = user.to_account_info();
let mut user_lamports = user_info.try_borrow_mut_lamports()?;
**user_lamports = (*user_lamports).saturating_add(amount);
```

キャンペーン側と全く同じ手順で、今度はユーザーの残高を増やす。`.saturating_sub`（引き算）の代わりに`.saturating_add`（足し算）を使っている点だけが異なる。

**全体のまとめ**:

```
withdraw(amount: 3 SOL)の場合:

キャンペーンアカウント              ユーザーアカウント
┌──────────────────┐             ┌──────────────────┐
│ lamports: 10 SOL │             │ lamports: 2 SOL  │
└──────────────────┘             └──────────────────┘
         │                                │
    saturating_sub(3)                saturating_add(3)
         │                                │
         ▼                                ▼
┌──────────────────┐             ┌──────────────────┐
│ lamports: 7 SOL  │             │ lamports: 5 SOL  │
└──────────────────┘             └──────────────────┘
```

このように、Solanaではlamports（SOLの数値）を直接書き換えることで送金を実現している。銀行の帳簿上で「Aの残高を減らし、Bの残高を増やす」のと同じ原理である。

**疑問: 減らす額と増やす額を自由に変えられるのでは？**

コードを見ると、`saturating_sub`と`saturating_add`に渡す値をプログラマが自由に書けるように見える。例えば「キャンペーンから3 SOL減らして、ユーザーに4 SOL増やす」と書けば、1 SOL憑から生み出せてしまうのではないか？

**答え: できない。Solanaランタイムが自動でチェックしている。**

プログラム内ではlamportsを自由に書き換えているように見えるが、トランザクション終了時にSolanaランタイム（バリデータ）が**全アカウントのlamports合計が変わっていないか**を自動検証する。

```
トランザクション開始時:
  キャンペーン: 10 SOL + ユーザー: 2 SOL = 合計 12 SOL

トランザクション終了時（正常）:
  キャンペーン:  7 SOL + ユーザー: 5 SOL = 合計 12 SOL  → OK

トランザクション終了時（不正な例）:
  キャンペーン:  7 SOL + ユーザー: 6 SOL = 合計 13 SOL  → ランタイムが拒否！
```

つまり、プログラム内で不整合な書き換えをしても、トランザクション全体の**lamports合計が変わった時点で自動的に失敗**する。これはプログラムのコードとは無関係に、Solanaの基盤レベルで強制されるルールである。

| チェック | 誰がやるか |
|---------|-----------|
| 権限チェック（admin == user?） | **プログラム**が行う（コードで書く） |
| Rent残高チェック | **プログラム**が行う（コードで書く） |
| lamports合計の整合性チェック | **Solanaランタイム**が自動で行う（コード不要） |

#### 4.4.3 `donate` 命令 — SOLの寄付

```rust
pub fn donate(ctx: Context<Donate>, amount: u64) -> Result<()> {
    let ix = anchor_lang::solana_program::system_instruction::transfer(
        &ctx.accounts.user.key(),
        &ctx.accounts.campaign.key(),
        amount
    );
    anchor_lang::solana_program::program::invoke(
        &ix,
        &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
    )?;
    (&mut ctx.accounts.campaign).amount_donated += amount;
    Ok(())
}
```

寄付の処理は2つのパートに分かれる。

**パート1: System Programを呼び出してSOLを送金する（CPI）**

```rust
let ix = anchor_lang::solana_program::system_instruction::transfer(
    &ctx.accounts.user.key(),   // 送金元
    &ctx.accounts.campaign.key(), // 送金先
    amount                       // 送金額
);
anchor_lang::solana_program::program::invoke(
    &ix,
    &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
)?;
```

| 要素 | 意味 |
|------|------|
| `system_instruction::transfer(...)` | 「AからBにSOLを送る」という命令データを作成する |
| `program::invoke(...)` | 作成した命令をSystem Programに渡して実行する（CPI） |
| CPI | Cross-Program Invocation（プログラム間呼び出し）の略。別のプログラムを呼び出すこと |

**`&ctx.accounts.user.key()` の `&` は「書き込みOK」ではない**:

`&`は「読み取り専用の参照」であり、`&mut`（書き込みOK）とは異なる。

| 記号 | 意味 | 読み | 書き |
|------|------|------|------|
| `&` | 参照（借りるだけ） | OK | NG |
| `&mut` | 可変参照（書き込みOK） | OK | OK |

ここでは`key()`が返した公開鍵（Pubkey）を**読むだけ**で十分。「誰から誰へ送るか」というアドレス情報を伝えるだけなので、書き換える必要がない。

```rust
&ctx.accounts.user.key()      // ユーザーの公開鍵を「見せるだけ」
&ctx.accounts.campaign.key()  // キャンペーンの公開鍵を「見せるだけ」
```

なぜ`&`が必要かというと、`transfer()`関数が「公開鍵そのもの」ではなく「公開鍵への参照」を受け取る設計になっているから。Rustでは大きなデータをコピーせずに渡すために、参照（`&`）を使うのが一般的である。

```
&なし: 公開鍵のデータ（32バイト）をまるごとコピーして渡す
&あり: 「ここにありますよ」という場所だけ伝える（効率的）
```

**`program::invoke()` の詳細解説**:

```rust
anchor_lang::solana_program::program::invoke(
    &ix,
    &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
)?;
```

この関数は**CPI（Cross-Program Invocation）**、つまり「プログラムの中から別のプログラムを呼び出す」処理である。

**第1引数: `&ix`（命令データ）**

直前の行で作った送金命令への参照。「何をするか」の指示書。

```rust
let ix = anchor_lang::solana_program::system_instruction::transfer(
    &ctx.accounts.user.key(),     // 送金元
    &ctx.accounts.campaign.key(), // 送金先
    amount                        // 送金額
);
```

この`ix`（instruction）の中には以下が入っている。

| 中身 | 値 |
|------|-----|
| 呼び出すプログラム | System Program |
| 送金元 | ユーザーの公開鍵 |
| 送金先 | キャンペーンの公開鍵 |
| 送金額 | amount（lamports） |

**第2引数: `&[...]`（関係するアカウント一覧）**

Solanaのルールとして、トランザクションで使う**すべてのアカウントの情報を渡す**必要がある。

```rust
&[
    ctx.accounts.user.to_account_info(),      // 送金元のアカウント情報
    ctx.accounts.campaign.to_account_info(),  // 送金先のアカウント情報
]
```

`&[...]`は「配列への参照」。この配列の中に、送金に関わる2つのアカウントの情報を入れている。

**なぜアカウント情報を別途渡す必要があるのか**:

`ix`の中に既に公開鍵（アドレス）が入っているのに、なぜ`AccountInfo`も渡すのか？

```
ix（命令）         → 「AからBに送って」という指示（アドレスだけ）
AccountInfo（情報） → アカウントの実データ（残高、データ、署名の有無など）
```

日常で例えると以下の通り。

| 引数 | 例え |
|------|------|
| `ix` | 銀行の振込用紙（「口座Aから口座Bに1万円」と書いてある） |
| `AccountInfo`の配列 | 口座AとBの通帳（実際の残高や口座名義が分かる） |

銀行は振込用紙だけでなく、通帳の中身も確認してから処理を実行する。Solanaも同じで、命令（何をするか）とアカウント情報（実データ）の両方がないと処理できない。

**`)?;` の意味**:

`invoke()`は失敗する可能性がある（残高不足、署名なし等）。`?`で「失敗したらエラーを返す」処理をしている（ステップ②で解説済み）。

**パート2: 寄付総額を更新する**

```rust
(&mut ctx.accounts.campaign).amount_donated += amount;
```

キャンペーンアカウントの`amount_donated`（寄付総額）に今回の寄付額を加算する。これによりキャンペーンに累計いくら寄付されたかを追跡できる。

### 4.5 `system_program` が必要な命令と不要な命令

ここが本日の学習で**最も重要な理解ポイント**である。

| 命令 | SOLの動かし方 | `system_program`が必要か |
|------|---------------|-------------------------|
| `create` | アカウント新規作成 | **必要**（System Programがアカウントを作る） |
| `withdraw` | プログラム内でlamportsを直接書き換え | **不要**（System Programを呼ばない） |
| `donate` | System Programの`transfer`をCPIで実行 | **必要**（呼び出すプログラムとして渡す） |

**なぜこの違いが生まれるのか?**

Solanaには「別のプログラムを`invoke`するなら、そのプログラムもアカウントとして渡す」というルールがある。

- `donate`ではSystem Programの`transfer`命令を`invoke`で呼び出している → System Programをアカウントとして渡す必要がある
- `withdraw`ではSystem Programを一切呼ばず、lamportsを直接書き換えている → System Programは不要

**なぜ`withdraw`はlamports直接操作を使うのか?**

`withdraw`でSOLを動かすのは「キャンペーンアカウント → ユーザー」の方向。キャンペーンアカウントはこのプログラムが所有しているため、プログラム内でlamportsを直接操作できる。一方`donate`では「ユーザー → キャンペーンアカウント」の方向で、ユーザーのSOLを動かすにはユーザーの署名が必要なため、署名検証を行うSystem Programを経由する必要がある。

### 4.6 アカウント制約（`#[derive(Accounts)]`）の定義

#### 構造体（struct）の基本

構造体は「複数のデータをひとまとめにする入れ物」。例えば人の情報を管理するなら：

```rust
// 構造体なし（バラバラ）
let name = "田中";
let age = 25;
let email = "tanaka@example.com";

// 構造体あり（ひとまとめ）
struct Person {
    name: String,
    age: u64,
    email: String,
}
```

| 部分 | 意味 |
|------|------|
| `struct` | 「構造体を定義します」というキーワード |
| `Person` | 構造体の名前（自由につけられる） |
| `{ ... }` | フィールド（中身）の定義 |
| `name: String` | 「nameという名前のフィールドで、型はString」 |

JavaScriptのオブジェクト、Pythonのクラスに近い概念である。

```
JavaScript:  const person = { name: "田中", age: 25 }
Python:      class Person: name, age
Rust:        struct Person { name: String, age: u64 }
```

#### `Create` 構造体 — PDAによるアカウント生成

```rust
#[derive(Accounts)]
pub struct Create<'info> {
    #[account(init, payer = user, space = 9000, seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()], bump)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}
```

この構造体には、通常のRust構造体にはないAnchor特有の記法がいくつかある。順番に解説する。

**`#[derive(Accounts)]` とは**:

`#[derive(...)]`は「この構造体に機能を自動で追加してね」とコンパイラに頼む仕組み（属性マクロ）。`Accounts`はAnchorが提供する機能で、以下を自動生成する。

```
#[derive(Accounts)] をつけると自動で行われること:
  ① トランザクションからアカウント情報を取り出す処理
  ② 各アカウントの制約（init, mut, seeds等）を検証する処理
  ③ アカウントデータをRustの型にデシリアライズする処理
```

もし`#[derive(Accounts)]`がなかったら、これらの処理を全部手書きする必要がある。

```rust
// #[derive(Accounts)]あり → Anchorが自動でやってくれる
#[derive(Accounts)]
pub struct Create<'info> { ... }

// #[derive(Accounts)]なし → 全部自分で書く（数十〜数百行必要）
// アカウントの取り出し、型チェック、権限チェック、PDA検証...
```

日常で例えると以下の通り。

| 場面 | 例え |
|------|------|
| `#[derive(Accounts)]`あり | 「確定申告ソフト」を使う。項目を入力すれば書類が自動作成される |
| `#[derive(Accounts)]`なし | 確定申告の書類を白紙から手書きする。計算も全部自分で行う |

**`<'info>` とは — ライフタイム**:

`<'info>`はRust特有の「ライフタイム」という仕組み。「このデータはいつまで有効か」をコンパイラに伝えるもの。

Solanaのアカウントデータはブロックチェーン上にあり、プログラム実行中に一時的にメモリに読み込まれる。`'info`は「この実行中だけ有効ですよ」という印。

```
プログラム実行開始
  ↓
  アカウントデータをメモリに読み込む ← ここから'infoが有効
  ↓
  create関数を実行する             ← 'infoの間はデータにアクセスできる
  ↓
  実行終了                         ← ここで'infoが終わり。メモリから解放
```

すべてのアカウント型（`Account<'info, Campaign>`、`Signer<'info>`、`Program<'info, System>`）に`'info`がついているのは、「全員が同じ実行期間中のデータを参照している」ことを示している。

**初学者へのアドバイス**: `<'info>`は「Anchorの構造体には必ずつけるもの」と覚えれば、当面は問題ない。深く理解するのはRustに慣れてからで十分。

**`#[account(mut)]` とは — アカウント制約**:

`#[account(...)]`は各フィールドにつける**アカウント制約**。Anchorが自動検証する条件を指定する。

`#[account(mut)]`の`mut`は`&mut`と同じ考え方で、「このアカウントのデータを変更する」という宣言。

```rust
#[account(mut)]        // このアカウントは変更される
pub user: Signer<'info>,

// mutなし（デフォルト）  // このアカウントは読み取りだけ
pub system_program: Program<'info, System>
```

なぜ明示的に宣言する必要があるかというと、Solanaランタイムが「本当に変更が必要なアカウントだけを書き込みロックする」ための情報として使うから。不要なアカウントまで書き込みロックすると、並列処理の効率が落ちる。

| 制約 | 意味 |
|------|------|
| `#[account(mut)]` | このアカウントのデータまたはSOL残高が変更される |
| 制約なし | このアカウントは読み取りだけ（変更しない） |

Create構造体の各フィールドを見ると以下の通り。

| フィールド | 制約 | 変更されるか | 理由 |
|-----------|------|-------------|------|
| `campaign` | `init`（新規作成 → 当然変更あり） | はい | データを書き込む |
| `user` | `mut` | はい | SOLが減る（アカウント作成費を支払うため） |
| `system_program` | なし | いいえ | 呼び出すだけ |

これらの知識を踏まえて、以下のアカウント制約の詳細を見ていく。

前回の電卓プログラムとの大きな違いは、`seeds`と`bump`の存在。これが**PDA（Program Derived Address）**である。

| 要素 | 意味 |
|------|------|
| `init` | 新しいアカウントを作成する |
| `payer = user` | アカウント作成の費用をユーザーが支払う |
| `space = 9000` | アカウントに9000バイトのデータ領域を確保する |
| `seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()]` | PDAのシード（アドレス生成の材料） |
| `bump` | PDAの衝突回避用パラメータ（Anchorが自動で探す） |

**PDA（Program Derived Address）とは?**

前回の電卓プログラムでは、テスト側で`Keypair.generate()`を使って毎回ランダムなアドレスを生成していた。PDAはこれとは異なり、**決まったルール（シード）からアドレスを自動的に計算する仕組み**。

```
PDAのアドレス = hash("CAMPAIGN_DEMO" + ユーザーの公開鍵 + bump + プログラムID)
```

PDAの利点:
- **再現性**: 同じシードからは常に同じアドレスが得られる。フロントエンドからもアドレスを計算できる
- **一意性**: 「CAMPAIGN_DEMO」+ ユーザーの公開鍵の組み合わせは1つだけ。同じユーザーが2つ目のキャンペーンを作ろうとするとエラーになる
- **キーペア不要**: テスト側で`Keypair.generate()`を呼ぶ必要がない。アドレスは自動導出される

**`space = 9000`について**:

前回の電卓プログラムでは`space = 264`だった。今回は`name`と`description`という2つのString型フィールドを持つため、より大きな領域が必要。9000バイトは余裕を持った値で、実運用ではより厳密に計算することが望ましい。

#### `Withdraw` 構造体

```rust
#[derive(Accounts)]
pub struct Withdraw<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
}
```

| 要素 | 意味 |
|------|------|
| `campaign` | 既存のキャンペーンアカウント。`mut`で書き込み可能（lamportsが減る） |
| `user` | 引き出しを行うユーザー（署名者）。`mut`で書き込み可能（lamportsが増える） |

`system_program`がないことに注目。前述の通り、`withdraw`はlamportsを直接操作するためSystem Programを呼ばない。

#### `Donate` 構造体

```rust
#[derive(Accounts)]
pub struct Donate<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}
```

`Withdraw`との違いは`system_program`の有無だけ。`donate`ではSystem Programの`transfer`をCPIで呼び出すため、`system_program`が必要。

### 4.7 状態構造体（`#[account]`）の定義

```rust
#[account]
pub struct Campaign {
    pub admin: Pubkey,
    pub name: String,
    pub description: String,
    pub amount_donated: u64
}
```

| フィールド | 型 | 用途 |
|-----------|-----|------|
| `admin` | `Pubkey` | キャンペーン管理者の公開鍵（32バイト） |
| `name` | `String` | キャンペーン名 |
| `description` | `String` | キャンペーンの説明文 |
| `amount_donated` | `u64` | 累計寄付額（lamports単位） |

前回の電卓プログラムの`Calculator`構造体と比べると、`admin`フィールド（`Pubkey`型）が新しい概念。これにより「誰がこのキャンペーンのオーナーか」をオンチェーンに記録し、権限管理に使う。

`u64`は64ビット**符号なし**整数型（0 〜 18,446,744,073,709,551,615）。SOLの額は常に0以上なので、前回の電卓で使った`i64`（符号あり）ではなく`u64`を使う。

## 7. Rust構文の補足（初学者向け）

### 7.1 前回からの復習

| 構文 | 意味 |
|------|------|
| `&mut` | 可変参照。データを変更するために必要 |
| `ctx.accounts.campaign` | コンテキストからアカウントにアクセスするドット記法 |
| `Ok(())` | 「成功、返す値なし」を意味するResult型の値 |
| `pub fn` | パブリック関数 |

### 7.2 今回新しく登場した構文

| 構文 | 意味 |
|------|------|
| `if ... { return Err(...); }` | 条件が満たされない場合にエラーを返して処理を中断する |
| `ProgramError::InvalidInstructionData` | Solana標準のエラー型。不正な命令データを示す |
| `ProgramError::InsufficientFunds` | Solana標準のエラー型。資金不足を示す |
| `.into()` | 型を変換する。ここでは`ProgramError`をAnchorの`Error`型に変換 |
| `Rent::get()?` | チェーンからRent情報を取得する。`?`はエラー時に即座にreturnする演算子 |
| `.try_borrow_mut_lamports()?` | lamports（SOL残高）を書き換え可能な参照として借りる |
| `**variable` | 二重デリファレンス。参照の参照を外して実際の値にアクセスする |
| `.saturating_sub(n)` | 安全な引き算。結果が負になる場合は0で止まる |
| `.saturating_add(n)` | 安全な足し算。オーバーフロー時は最大値で止まる |
| `b"CAMPAIGN_DEMO".as_ref()` | 文字列をバイト列の参照に変換する。PDAのシードとして使用 |
| `user.key().as_ref()` | 公開鍵をバイト列の参照に変換する。PDAのシードとして使用 |

### 7.3 `saturating_sub` / `saturating_add` とは

通常の引き算（`-`）や足し算（`+`）では、結果が型の範囲を超えるとパニック（プログラムの異常終了）が発生する。`saturating_sub`と`saturating_add`は、範囲を超えた場合に**安全に止まる**演算。

```
通常の引き算:   5 - 10 = パニック！（u64では負の数を表現できない）
saturating_sub: 5.saturating_sub(10) = 0（0で止まる）

通常の足し算:   u64::MAX + 1 = パニック！（オーバーフロー）
saturating_add: u64::MAX.saturating_add(1) = u64::MAX（最大値で止まる）
```

### 7.4 `u64` とは — Rustの整数型

`u64`はこのプログラムで`amount`（寄付額・引き出し額）や`amount_donated`（寄付総額）に使われている型である。

**名前の意味**:

```
u   → unsigned（符号なし = マイナスがない）
64  → 64ビット（メモリ上で64桁の2進数を使う）
```

**`u64`と`i64`の違い**:

| 型 | 範囲 | 用途 |
|----|------|------|
| `u64`（符号なし） | 0 〜 18,446,744,073,709,551,615 | SOLの額など、常に0以上の値 |
| `i64`（符号あり） | -9,223,372,036,854,775,808 〜 9,223,372,036,854,775,807 | マイナスもあり得る値（前回の電卓で使用） |

**なぜ`u64`を使うのか**:

SOLの額はマイナスになることがない。「-5 SOL寄付する」は意味がないため、マイナスを扱えない`u64`を使うことで、プログラムレベルで負の値を防いでいる。

```rust
// u64 の場合
let amount: u64 = 100;    // OK
let amount: u64 = -100;   // コンパイルエラー！負の値は入れられない

// i64 の場合（前回の電卓）
let result: i64 = 100;    // OK
let result: i64 = -100;   // OK（引き算の結果がマイナスになる場合がある）
```

**Rustの整数型の一覧**:

| 型 | ビット数 | 符号 | 範囲 |
|----|---------|------|------|
| `u8` | 8 | なし | 0 〜 255 |
| `u16` | 16 | なし | 0 〜 65,535 |
| `u32` | 32 | なし | 0 〜 約42億 |
| `u64` | 64 | なし | 0 〜 約1844京 |
| `i8` | 8 | あり | -128 〜 127 |
| `i16` | 16 | あり | -32,768 〜 32,767 |
| `i32` | 32 | あり | -約21億 〜 約21億 |
| `i64` | 64 | あり | -約922京 〜 約922京 |

Solanaでは`u64`が最もよく使われる。SOLの最小単位（lamport）が整数で、1 SOL = 1,000,000,000 lamportsのため、`u64`なら約184億SOLまで扱える。

## 8. 学習の成果

### 8.1 習得した内容

1. クラウドファンディングプログラムの3命令（create / withdraw / donate）の実装

### 8.2 重要な理解ポイント

- **PDAはシードからアドレスを自動計算する仕組み**で、`Keypair.generate()`とは異なりフロントエンドからもアドレスを再現できる
- **SOLの移動方法は「誰の資金を動かすか」で決まる**: ユーザーの資金 → CPI（System Program必要）、プログラム所有アカウントの資金 → lamports直接操作（System Program不要）
- **`system_program`が必要かどうかは「System Programを呼び出すかどうか」で決まる**
- **Rent免除残高を考慮しないと、引き出し時にアカウントが消えてしまう可能性がある**
- **`admin`フィールドによる権限管理は、DeFiやNFTプロジェクトでも頻出するパターン**
