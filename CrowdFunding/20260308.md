**日付**: 2026年3月8日
**学習内容**: Anchorによるクラウドファンディングプログラムの実装、PDA（Program Derived Address）によるアカウント生成、SOLの送金（CPI）と直接lamports操作の違い、`system_program`の要否の理解

## 1. 学習の全体像

本日の学習では、前回の電卓プログラムから一歩進み、実用的な**クラウドファンディングプログラム**をAnchorで実装した。
特に、次の3点を重点的に理解した。

- キャンペーン（Campaign）の作成・寄付・引き出しという3つの命令の実装
- PDA（Program Derived Address）を使ったアカウントアドレスの自動導出
- SOLを動かす2つの方法（CPI経由 vs lamports直接操作）と`system_program`が必要になる条件

## 2. プロジェクトのセットアップ

### 2.1 Anchorプロジェクトの初期化

```bash
anchor init crowdfunding
cd crowdfunding
anchor build
```

`anchor init crowdfunding`で新しいAnchorプロジェクトが生成される。
前回の電卓プロジェクトと同じ手順だが、今回のプログラム名は`crowdfunding`になる。

### 2.2 Anchorバージョンの確認

`programs/crowdfunding/Cargo.toml`に以下の記述がある。

```toml
[dependencies]
anchor-lang = "0.32.1"
```

この行でAnchorフレームワークのバージョンを指定する。プロジェクト内のRustコードは、このバージョンのAnchor APIに基づいて書く。

### 2.3 IDL（Interface Definition Language）の自動生成

`anchor build`を実行すると、`target/idl/crowdfunding.json`にプログラムのIDLが自動生成される。

IDLとは「プログラムの取扱説明書」のようなもので、以下の情報がJSON形式で記述される。

- どんな命令（instruction）があるか
- 各命令にどんな引数が必要か
- 各命令にどんなアカウントを渡すか
- アカウントのデータ構造（型情報）

フロントエンドやテストコードは、このIDLを読み込むことで、プログラムの命令を型安全に呼び出せるようになる。

## 4. `lib.rs` のコード詳細解説

対象: `programs/crowdfunding/src/lib.rs`

### 4.1 全体コード

```rust
use anchor_lang::prelude::*;

declare_id!("FiPBwdr9QMkpeJuXhm5Dr2tzxN8DbYod2g6Q9eEKvYdL");

#[program]
pub mod crowdfunding {
    use super::*;

    pub fn create(ctx: Context<Create>, name: String, description: String) -> Result<()> {
        let campaign = &mut ctx.accounts.campaign;
        campaign.name = name;
        campaign.description = description;
        campaign.amount_donated = 0;
        campaign.admin = ctx.accounts.user.key();
        Ok(())
    }

    pub fn withdraw(ctx: Context<Withdraw>, amount: u64) -> Result<()> {
        let campaign = &mut ctx.accounts.campaign;
        let user = &mut ctx.accounts.user;
        if campaign.admin != user.key() {
            return Err(ProgramError::InvalidInstructionData.into());
        }
        let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
        if campaign.to_account_info().lamports() - rent_balance < amount {
            return Err(ProgramError::InsufficientFunds.into());
        }
        let campaign_info = campaign.to_account_info();
        let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
        **campaign_lamports = (*campaign_lamports).saturating_sub(amount);
        let user_info = user.to_account_info();
        let mut user_lamports = user_info.try_borrow_mut_lamports()?;
        **user_lamports = (*user_lamports).saturating_add(amount);
        Ok(())
    }

    pub fn donate(ctx: Context<Donate>, amount: u64) -> Result<()> {
        let ix = anchor_lang::solana_program::system_instruction::transfer(
            &ctx.accounts.user.key(),
            &ctx.accounts.campaign.key(),
            amount
        );
        anchor_lang::solana_program::program::invoke(
            &ix,
            &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
        )?;
        (&mut ctx.accounts.campaign).amount_donated += amount;
        Ok(())
    }
}

#[derive(Accounts)]
pub struct Create<'info> {
    #[account(init, payer = user, space = 9000, seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()], bump)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}

#[derive(Accounts)]
pub struct Withdraw<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
}

#[derive(Accounts)]
pub struct Donate<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}

#[account]
pub struct Campaign {
    pub admin: Pubkey,
    pub name: String,
    pub description: String,
    pub amount_donated: u64
}
```

#### 同じアドレスにデプロイすると上書きされるのか？

**同じアドレスにデプロイすると、既存のプログラムは上書きされる。** ただし、いくつか重要な条件がある。

**上書きできる条件**:

| 条件 | 説明 |
|------|------|
| アップグレード権限を持っている | デプロイ時に使ったウォレット（Upgrade Authority）と同じ秘密鍵でないと上書きできない |
| プログラムがアップグレード可能な状態 | デフォルトでは`upgradeable`（上書き可能）でデプロイされる |

つまり、自分の秘密鍵（`~/.config/solana/id.json`）を持っている人だけが上書きできる。他人が同じアドレスに勝手にデプロイすることはできない。

**ネットワークごとの独立性**:

```
localnet  → FiPBwdr9Q... のプログラム（ローカルだけに存在）
devnet    → FiPBwdr9Q... のプログラム（devnetだけに存在）
mainnet   → FiPBwdr9Q... のプログラム（mainnetだけに存在）
```

各ネットワークは完全に独立しており、localnetにデプロイしてもdevnetやmainnetには影響しない。

**上書きを禁止する方法**:

本番運用時に「もう二度とコードを変更しない」と決めた場合、以下のコマンドでアップグレード権限を放棄できる。

```bash
solana program set-upgrade-authority <PROGRAM_ID> --final
```

これを実行すると、誰もプログラムを上書きできなくなる（不可逆）。

**Ethereumとの比較**:

| 観点 | Ethereum | Solana |
|------|----------|--------|
| デフォルト | **イミュータブル**（上書き不可） | **アップグレード可能**（上書き可能） |
| 上書きしたい場合 | Proxyパターン等の工夫が必要 | そのまま`solana program deploy`するだけ |
| 上書き禁止 | デフォルトでそうなっている | `--final`フラグで明示的に設定 |

Solanaの方が「開発中は気軽にバグ修正できる」設計になっている。

**ユーザー側からプログラムの状態を確認する方法**:

誰でも以下のコマンドでプログラムの状態を確認できる。

```bash
solana program show <PROGRAM_ID>
```

出力例:

```
Program Id: FiPBwdr9QMkpeJuXhm5Dr2tzxN8DbYod2g6Q9eEKvYdL
Owner: BPFLoaderUpgradeab1e11111111111111111111111
ProgramData Address: 3Vg9...xxxx
Authority: 7nYB...yyyy        ← Upgrade Authorityの公開鍵
Last Deployed In Slot: 12345
Data Length: 200000 bytes
```

| フィールド | 意味 |
|-----------|------|
| `Authority` | アップグレード権限を持つウォレットの公開鍵。**ここが表示されていれば上書き可能** |
| `Authority: None` | アップグレード権限が放棄済み。**誰も上書きできない（イミュータブル）** |

ユーザーは`Authority`の有無を見るだけで「このプログラムは今後変更される可能性があるか」を判断できる。DeFiプロジェクトなどでは、`Authority: None`であることが信頼性の指標として重視される。開発者がいつでもコードを書き換えられる状態は、ユーザーにとってリスクになるためである。

### 4.4 `#[program]` ブロック — 3つの命令の定義

このプログラムには3つの命令がある。

| 命令 | 役割 | 引数 |
|------|------|------|
| `create` | キャンペーンを新規作成する | `name`（名前）、`description`（説明） |
| `withdraw` | キャンペーンからSOLを引き出す | `amount`（引き出し額） |
| `donate` | キャンペーンにSOLを寄付する | `amount`（寄付額） |

#### 4.4.1 `create` 命令 — キャンペーンの作成

```rust
pub fn create(ctx: Context<Create>, name: String, description: String) -> Result<()> {
    let campaign = &mut ctx.accounts.campaign;
    campaign.name = name;
    campaign.description = description;
    campaign.amount_donated = 0;
    campaign.admin = ctx.accounts.user.key();
    Ok(())
}
```

この命令は新しいクラウドファンディングのキャンペーンを作成する。

**注意: `create`はデプロイ時に自動実行されない**:

Ethereumでは`constructor`がデプロイ時に1回だけ自動実行され、初期化処理を行う。しかしSolanaにはその仕組みがない。全ての命令は外部から明示的にトランザクションを送って呼び出す必要がある。

| ステップ | 何が起こるか |
|---------|-------------|
| `anchor build` | Rustコードをバイナリ（`.so`）にコンパイルする |
| `anchor deploy` | バイナリをSolanaネットワークにアップロードする。**プログラムが使える状態になるだけ** |
| `create`命令の呼び出し | ユーザー（またはテスト）が**明示的にトランザクションを送って**初めてキャンペーンが作成される |

つまりデプロイ直後のプログラムは「命令を受け付ける準備ができた状態」であり、キャンペーンなどのデータはまだ存在しない。

| 行 | 処理内容 |
|----|----------|
| `let campaign = &mut ctx.accounts.campaign;` | コンテキストから`campaign`アカウントを可変参照（書き込み可能な形）で取得する（後述） |
| `campaign.name = name;` | 引数で受け取ったキャンペーン名を保存する |
| `campaign.description = description;` | 引数で受け取ったキャンペーン説明を保存する |
| `campaign.amount_donated = 0;` | 寄付総額を0で初期化する |
| `campaign.admin = ctx.accounts.user.key();` | キャンペーンを作った人（署名者）のアドレスを管理者として記録する |
| `Ok(())` | 「処理が成功しました」を返す |

**ポイント**: `ctx.accounts.user.key()`でトランザクションの署名者（=キャンペーン作成者）の公開鍵を取得し、`admin`フィールドに保存している。これにより、後の`withdraw`命令で「管理者本人だけが引き出しできる」という権限チェックが可能になる。

**`let campaign = &mut ctx.accounts.campaign;` の詳細解説**:

この1行は初見では分かりにくいため、分解して説明する。

まず`ctx`（Context）とは何か。トランザクションが送られると、Anchorは命令に必要なアカウント群を自動的に検証・整理し、1つのオブジェクトにまとめて関数に渡す。これが`ctx`である。

```
ユーザーがトランザクション送信
    ↓
Anchorが自動でアカウントを検証・整理
    ↓
ctx というオブジェクトにまとめて関数に渡す
    ↓
ctx.accounts.campaign でアクセスできる
```

つまり`ctx`は「この命令で使うアカウントが全部入った箱」である。

次に`&mut`がなぜ必要かを説明する。Rustでは変数はデフォルトで**読み取り専用**である。データを変更するには`&mut`（可変参照）が必要になる。

```rust
// これだと読み取りしかできない（変更不可）
let campaign = ctx.accounts.campaign;
campaign.name = name;  // エラー！変更できない

// &mut をつけると書き込み可能になる
let campaign = &mut ctx.accounts.campaign;
campaign.name = name;  // OK！変更できる
```

現実世界に例えると以下のようになる。

| 操作 | 例え |
|------|------|
| `&`（参照） | 図書館の本を**閲覧**する。読めるが書き込めない |
| `&mut`（可変参照） | 図書館の本を**借りて持ち帰る**。書き込みもできる |

各部分の意味をまとめると以下の通り。

| 部分 | 意味 |
|------|------|
| `ctx` | 命令に渡されたアカウント群が入った箱 |
| `.accounts` | その中のアカウント一覧にアクセス |
| `.campaign` | `Create`構造体で定義した`campaign`アカウントを取り出す |
| `&mut` | 「このアカウントのデータを変更しますよ」という宣言 |
| `let campaign =` | 取り出したものを`campaign`という変数名で使えるようにする |

この後の行で`campaign.name = name;`のように値を書き込めるのは、`&mut`で「変更可能」として取り出したからである。

#### 4.4.2 `withdraw` 命令 — SOLの引き出し

```rust
pub fn withdraw(ctx: Context<Withdraw>, amount: u64) -> Result<()> {
    let campaign = &mut ctx.accounts.campaign;
    let user = &mut ctx.accounts.user;
    // ① 権限チェック: 管理者本人か？
    if campaign.admin != user.key() {
        return Err(ProgramError::InvalidInstructionData.into());
    }
    // ② 残高チェック: 引き出し可能な額があるか？
    let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
    if campaign.to_account_info().lamports() - rent_balance < amount {
        return Err(ProgramError::InsufficientFunds.into());
    }
    // ③ lamportsを直接操作してSOLを移動する
    let campaign_info = campaign.to_account_info();
    let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
    **campaign_lamports = (*campaign_lamports).saturating_sub(amount);
    let user_info = user.to_account_info();
    let mut user_lamports = user_info.try_borrow_mut_lamports()?;
    **user_lamports = (*user_lamports).saturating_add(amount);
    Ok(())
}
```

この命令はキャンペーンに集まったSOLを管理者が引き出すためのもので、3つのステップで構成される。

**ステップ①: 権限チェック**

```rust
if campaign.admin != user.key() {
    return Err(ProgramError::InvalidInstructionData.into());
}
```

キャンペーンの`admin`（作成時に記録された管理者アドレス）と、今回の呼び出し者（`user`）が一致するか確認する。一致しなければエラーを返して処理を中断する。これにより**管理者以外の人はSOLを引き出せない**。

**ステップ②: 残高チェック（Rent免除の考慮）**

```rust
let rent_balance = Rent::get()?.minimum_balance(campaign.to_account_info().data_len());
if campaign.to_account_info().lamports() - rent_balance < amount {
    return Err(ProgramError::InsufficientFunds.into());
}
```

Solanaでは、アカウントのデータを維持するために**Rent（家賃）**と呼ばれる最低残高が必要。この最低残高を下回るとアカウントが削除されてしまう。

| 要素 | 意味 |
|------|------|
| `Rent::get()?` | 現在のRentパラメータをチェーンから取得する |
| `.minimum_balance(data_len)` | アカウントのデータサイズに応じた最低必要残高を計算する |
| `campaign.to_account_info().lamports()` | キャンペーンアカウントの現在のSOL残高（lamports単位）を取得する |

つまり「現在の残高 - Rent免除に必要な額」が引き出し額より少なければ、資金不足エラーを返す。

**ステップ③: lamportsの直接操作**

```rust
let campaign_info = campaign.to_account_info();
let mut campaign_lamports = campaign_info.try_borrow_mut_lamports()?;
**campaign_lamports = (*campaign_lamports).saturating_sub(amount);
let user_info = user.to_account_info();
let mut user_lamports = user_info.try_borrow_mut_lamports()?;
**user_lamports = (*user_lamports).saturating_add(amount);
```

ここがSOLの実際の移動処理。1行ずつ解説する。

| 行 | 処理内容 |
|----|----------|
| `campaign.to_account_info()` | campaignアカウントの低レベル情報を取得する |
| `.try_borrow_mut_lamports()?` | lamports（SOL残高）を書き換え可能な参照として借りる |
| `**campaign_lamports` | 二重参照を外して実際の数値にアクセスする |
| `.saturating_sub(amount)` | 引き算する。結果が負になる場合は0で止まる（安全な引き算） |
| `.saturating_add(amount)` | 足し算する。オーバーフローする場合は最大値で止まる（安全な足し算） |

**`**`（二重デリファレンス）の解説**: Rustでは`try_borrow_mut_lamports()`は「参照の参照」（`&mut &mut u64`のようなもの）を返す。実際の数値にアクセスするには`*`を2回使って参照を外す必要がある。イメージとしては「箱の中の箱の中の数値」を取り出すようなもの。

#### 4.4.3 `donate` 命令 — SOLの寄付

```rust
pub fn donate(ctx: Context<Donate>, amount: u64) -> Result<()> {
    let ix = anchor_lang::solana_program::system_instruction::transfer(
        &ctx.accounts.user.key(),
        &ctx.accounts.campaign.key(),
        amount
    );
    anchor_lang::solana_program::program::invoke(
        &ix,
        &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
    )?;
    (&mut ctx.accounts.campaign).amount_donated += amount;
    Ok(())
}
```

寄付の処理は2つのパートに分かれる。

**パート1: System Programを呼び出してSOLを送金する（CPI）**

```rust
let ix = anchor_lang::solana_program::system_instruction::transfer(
    &ctx.accounts.user.key(),   // 送金元
    &ctx.accounts.campaign.key(), // 送金先
    amount                       // 送金額
);
anchor_lang::solana_program::program::invoke(
    &ix,
    &[ctx.accounts.user.to_account_info(), ctx.accounts.campaign.to_account_info()],
)?;
```

| 要素 | 意味 |
|------|------|
| `system_instruction::transfer(...)` | 「AからBにSOLを送る」という命令データを作成する |
| `program::invoke(...)` | 作成した命令をSystem Programに渡して実行する（CPI） |
| CPI | Cross-Program Invocation（プログラム間呼び出し）の略。別のプログラムを呼び出すこと |

**パート2: 寄付総額を更新する**

```rust
(&mut ctx.accounts.campaign).amount_donated += amount;
```

キャンペーンアカウントの`amount_donated`（寄付総額）に今回の寄付額を加算する。これによりキャンペーンに累計いくら寄付されたかを追跡できる。

### 4.5 `system_program` が必要な命令と不要な命令

ここが本日の学習で**最も重要な理解ポイント**である。

| 命令 | SOLの動かし方 | `system_program`が必要か |
|------|---------------|-------------------------|
| `create` | アカウント新規作成 | **必要**（System Programがアカウントを作る） |
| `withdraw` | プログラム内でlamportsを直接書き換え | **不要**（System Programを呼ばない） |
| `donate` | System Programの`transfer`をCPIで実行 | **必要**（呼び出すプログラムとして渡す） |

**なぜこの違いが生まれるのか?**

Solanaには「別のプログラムを`invoke`するなら、そのプログラムもアカウントとして渡す」というルールがある。

- `donate`ではSystem Programの`transfer`命令を`invoke`で呼び出している → System Programをアカウントとして渡す必要がある
- `withdraw`ではSystem Programを一切呼ばず、lamportsを直接書き換えている → System Programは不要

**なぜ`withdraw`はlamports直接操作を使うのか?**

`withdraw`でSOLを動かすのは「キャンペーンアカウント → ユーザー」の方向。キャンペーンアカウントはこのプログラムが所有しているため、プログラム内でlamportsを直接操作できる。一方`donate`では「ユーザー → キャンペーンアカウント」の方向で、ユーザーのSOLを動かすにはユーザーの署名が必要なため、署名検証を行うSystem Programを経由する必要がある。

### 4.6 アカウント制約（`#[derive(Accounts)]`）の定義

#### `Create` 構造体 — PDAによるアカウント生成

```rust
#[derive(Accounts)]
pub struct Create<'info> {
    #[account(init, payer = user, space = 9000, seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()], bump)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}
```

前回の電卓プログラムとの大きな違いは、`seeds`と`bump`の存在。これが**PDA（Program Derived Address）**である。

| 要素 | 意味 |
|------|------|
| `init` | 新しいアカウントを作成する |
| `payer = user` | アカウント作成の費用をユーザーが支払う |
| `space = 9000` | アカウントに9000バイトのデータ領域を確保する |
| `seeds=[b"CAMPAIGN_DEMO".as_ref(), user.key().as_ref()]` | PDAのシード（アドレス生成の材料） |
| `bump` | PDAの衝突回避用パラメータ（Anchorが自動で探す） |

**PDA（Program Derived Address）とは?**

前回の電卓プログラムでは、テスト側で`Keypair.generate()`を使って毎回ランダムなアドレスを生成していた。PDAはこれとは異なり、**決まったルール（シード）からアドレスを自動的に計算する仕組み**。

```
PDAのアドレス = hash("CAMPAIGN_DEMO" + ユーザーの公開鍵 + bump + プログラムID)
```

PDAの利点:
- **再現性**: 同じシードからは常に同じアドレスが得られる。フロントエンドからもアドレスを計算できる
- **一意性**: 「CAMPAIGN_DEMO」+ ユーザーの公開鍵の組み合わせは1つだけ。同じユーザーが2つ目のキャンペーンを作ろうとするとエラーになる
- **キーペア不要**: テスト側で`Keypair.generate()`を呼ぶ必要がない。アドレスは自動導出される

**`space = 9000`について**:

前回の電卓プログラムでは`space = 264`だった。今回は`name`と`description`という2つのString型フィールドを持つため、より大きな領域が必要。9000バイトは余裕を持った値で、実運用ではより厳密に計算することが望ましい。

#### `Withdraw` 構造体

```rust
#[derive(Accounts)]
pub struct Withdraw<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
}
```

| 要素 | 意味 |
|------|------|
| `campaign` | 既存のキャンペーンアカウント。`mut`で書き込み可能（lamportsが減る） |
| `user` | 引き出しを行うユーザー（署名者）。`mut`で書き込み可能（lamportsが増える） |

`system_program`がないことに注目。前述の通り、`withdraw`はlamportsを直接操作するためSystem Programを呼ばない。

#### `Donate` 構造体

```rust
#[derive(Accounts)]
pub struct Donate<'info> {
    #[account(mut)]
    pub campaign: Account<'info, Campaign>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>
}
```

`Withdraw`との違いは`system_program`の有無だけ。`donate`ではSystem Programの`transfer`をCPIで呼び出すため、`system_program`が必要。

### 4.7 状態構造体（`#[account]`）の定義

```rust
#[account]
pub struct Campaign {
    pub admin: Pubkey,
    pub name: String,
    pub description: String,
    pub amount_donated: u64
}
```

| フィールド | 型 | 用途 |
|-----------|-----|------|
| `admin` | `Pubkey` | キャンペーン管理者の公開鍵（32バイト） |
| `name` | `String` | キャンペーン名 |
| `description` | `String` | キャンペーンの説明文 |
| `amount_donated` | `u64` | 累計寄付額（lamports単位） |

前回の電卓プログラムの`Calculator`構造体と比べると、`admin`フィールド（`Pubkey`型）が新しい概念。これにより「誰がこのキャンペーンのオーナーか」をオンチェーンに記録し、権限管理に使う。

`u64`は64ビット**符号なし**整数型（0 〜 18,446,744,073,709,551,615）。SOLの額は常に0以上なので、前回の電卓で使った`i64`（符号あり）ではなく`u64`を使う。

## 7. Rust構文の補足（初学者向け）

### 7.1 前回からの復習

| 構文 | 意味 |
|------|------|
| `&mut` | 可変参照。データを変更するために必要 |
| `ctx.accounts.campaign` | コンテキストからアカウントにアクセスするドット記法 |
| `Ok(())` | 「成功、返す値なし」を意味するResult型の値 |
| `pub fn` | パブリック関数 |

### 7.2 今回新しく登場した構文

| 構文 | 意味 |
|------|------|
| `if ... { return Err(...); }` | 条件が満たされない場合にエラーを返して処理を中断する |
| `ProgramError::InvalidInstructionData` | Solana標準のエラー型。不正な命令データを示す |
| `ProgramError::InsufficientFunds` | Solana標準のエラー型。資金不足を示す |
| `.into()` | 型を変換する。ここでは`ProgramError`をAnchorの`Error`型に変換 |
| `Rent::get()?` | チェーンからRent情報を取得する。`?`はエラー時に即座にreturnする演算子 |
| `.try_borrow_mut_lamports()?` | lamports（SOL残高）を書き換え可能な参照として借りる |
| `**variable` | 二重デリファレンス。参照の参照を外して実際の値にアクセスする |
| `.saturating_sub(n)` | 安全な引き算。結果が負になる場合は0で止まる |
| `.saturating_add(n)` | 安全な足し算。オーバーフロー時は最大値で止まる |
| `b"CAMPAIGN_DEMO".as_ref()` | 文字列をバイト列の参照に変換する。PDAのシードとして使用 |
| `user.key().as_ref()` | 公開鍵をバイト列の参照に変換する。PDAのシードとして使用 |

### 7.3 `saturating_sub` / `saturating_add` とは

通常の引き算（`-`）や足し算（`+`）では、結果が型の範囲を超えるとパニック（プログラムの異常終了）が発生する。`saturating_sub`と`saturating_add`は、範囲を超えた場合に**安全に止まる**演算。

```
通常の引き算:   5 - 10 = パニック！（u64では負の数を表現できない）
saturating_sub: 5.saturating_sub(10) = 0（0で止まる）

通常の足し算:   u64::MAX + 1 = パニック！（オーバーフロー）
saturating_add: u64::MAX.saturating_add(1) = u64::MAX（最大値で止まる）
```

### 8.2 重要な理解ポイント

- **PDAはシードからアドレスを自動計算する仕組み**で、`Keypair.generate()`とは異なりフロントエンドからもアドレスを再現できる
- **SOLの移動方法は「誰の資金を動かすか」で決まる**: ユーザーの資金 → CPI（System Program必要）、プログラム所有アカウントの資金 → lamports直接操作（System Program不要）
- **`system_program`が必要かどうかは「System Programを呼び出すかどうか」で決まる**
- **Rent免除残高を考慮しないと、引き出し時にアカウントが消えてしまう可能性がある**
- **`admin`フィールドによる権限管理は、DeFiやNFTプロジェクトでも頻出するパターン**
